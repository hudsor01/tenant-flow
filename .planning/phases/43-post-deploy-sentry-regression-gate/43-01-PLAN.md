```yaml
# Post-Deploy Sentry Regression Gate (Phase 43 / DEPLOY-01, DEPLOY-02)
#
# What this does:
#   After a production Vercel deploy, wait an observation window, query Sentry
#   against the deployed release SHA, compare issue/error-rate profile against
#   the previous release's baseline artifact, and fail the workflow + open a
#   GitHub issue if any threshold is breached.
#
# Triggers:
#   - repository_dispatch: vercel.deployment.success | vercel.deployment.promoted
#     (Vercel's canonical 2026 integration — requires Vercel project Git settings
#      to have deployment events enabled; verify Project → Settings → Git in Vercel.)
#   - workflow_dispatch with input `release_sha` for manual smoke testing / re-runs.
#
# Smoke test (run after merging to main):
#   gh workflow run post-deploy-sentry-gate.yml -f release_sha=<known-good-shipped-sha>
#   gh run watch
#   # Expect green exit, PASS summary, and artifact `sentry-baseline-<sha>` uploaded.
#
# Required repo secrets:
#   SENTRY_AUTH_TOKEN  — Sentry Organization Token (Settings → Developer Settings
#                        → Organization Tokens). Scopes needed: project:read, event:read.
#   SENTRY_ORG         — Sentry org slug (matches next.config.ts:50)
#   SENTRY_PROJECT     — Sentry project slug (matches next.config.ts:51)
#
# Required repo variables (all tunable without code change via `gh variable set`):
#   SENTRY_GATE_NEW_ISSUE_EVENT_THRESHOLD  (default 10)
#   SENTRY_GATE_ERROR_RATE_DELTA_PCT       (default 1.0)
#   SENTRY_GATE_OBSERVATION_WINDOW_MINUTES (default 30)
#
# Threshold tuning: after 5+ production deploys, review opened issues vs false
# positives. Tighten or relax via `gh variable set SENTRY_GATE_<NAME> --body <value>`.
#
# Out of scope (per 43-DECISIONS.md): Vercel rollback, Slack/PagerDuty webhooks,
# rolling-median baseline, preview-deploy gating, performance (p95) threshold enforcement.

name: Post-Deploy Sentry Regression Gate

on:
  repository_dispatch:
    types:
      - vercel.deployment.success
      - vercel.deployment.promoted
  workflow_dispatch:
    inputs:
      release_sha:
        description: 'Git SHA to audit (for smoke tests or manual re-runs against a known-good release)'
        required: true
        type: string

concurrency:
  group: sentry-gate-${{ github.event.client_payload.git.sha || inputs.release_sha }}
  cancel-in-progress: false

permissions:
  contents: read
  actions: read
  issues: write

jobs:
  gate:
    if: github.event_name == 'workflow_dispatch' || github.event.client_payload.environment == 'production'
    runs-on: ubuntu-latest
    timeout-minutes: 60
    env:
      SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
      SENTRY_ORG: ${{ secrets.SENTRY_ORG }}
      SENTRY_PROJECT: ${{ secrets.SENTRY_PROJECT }}
      NEW_ISSUE_EVENT_THRESHOLD: ${{ vars.SENTRY_GATE_NEW_ISSUE_EVENT_THRESHOLD || '10' }}
      ERROR_RATE_DELTA_PCT: ${{ vars.SENTRY_GATE_ERROR_RATE_DELTA_PCT || '1.0' }}
      OBSERVATION_WINDOW_MINUTES: ${{ vars.SENTRY_GATE_OBSERVATION_WINDOW_MINUTES || '30' }}
      GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
    steps:
      - name: Check required secrets
        run: |
          missing=()
          [ -z "${SENTRY_AUTH_TOKEN:-}" ] && missing+=(SENTRY_AUTH_TOKEN)
          [ -z "${SENTRY_ORG:-}" ]        && missing+=(SENTRY_ORG)
          [ -z "${SENTRY_PROJECT:-}" ]    && missing+=(SENTRY_PROJECT)
          if [ ${#missing[@]} -gt 0 ]; then
            echo "::error::Missing required secrets: ${missing[*]}"
            exit 1
          fi

      - name: Resolve release SHA
        id: release
        env:
          CLIENT_PAYLOAD_SHA: ${{ github.event.client_payload.git.sha }}
          INPUT_RELEASE_SHA: ${{ inputs.release_sha }}
          EVENT_NAME: ${{ github.event_name }}
        run: |
          if [ "${EVENT_NAME}" = "workflow_dispatch" ]; then
            SHA="${INPUT_RELEASE_SHA}"
          else
            SHA="${CLIENT_PAYLOAD_SHA}"
          fi
          # T-43-03 mitigation: regex-validate before passing to any curl URL path.
          if ! echo "${SHA}" | grep -Eq '^[a-f0-9]{40}$'; then
            echo "::error::Invalid release SHA format: ${SHA}. Expected 40-char hex git SHA."
            exit 1
          fi
          SHORT="${SHA:0:7}"
          echo "sha=${SHA}" >> "$GITHUB_OUTPUT"
          echo "short_sha=${SHORT}" >> "$GITHUB_OUTPUT"
          echo "Release SHA: ${SHA} (short: ${SHORT})"

      - name: Observation window
        run: |
          echo "Sleeping ${OBSERVATION_WINDOW_MINUTES} minute(s) for Sentry to receive error/session signal..."
          sleep "${OBSERVATION_WINDOW_MINUTES}m"

      # Sentry query, baseline fetch, gate evaluation, issue creation, artifact upload
      # steps are added in Tasks 2-5 below. Task 1 stops here.
```

```yaml
      - name: Confirm release exists in Sentry
        env:
          RELEASE_SHA: ${{ steps.release.outputs.sha }}
        run: |
          STATUS=$(curl -sS -o /dev/null -w "%{http_code}" \
            "https://sentry.io/api/0/organizations/${SENTRY_ORG}/releases/${RELEASE_SHA}/" \
            -H "Authorization: Bearer ${SENTRY_AUTH_TOKEN}")
          if [ "${STATUS}" = "404" ]; then
            echo "::error::Sentry release ${RELEASE_SHA} not found. Check that @sentry/nextjs setCommits succeeded in the Vercel build. See next.config.ts:65-70."
            exit 1
          fi
          if [ "${STATUS}" != "200" ]; then
            echo "::error::Sentry releases API returned HTTP ${STATUS}. Check SENTRY_AUTH_TOKEN scopes (project:read required)."
            exit 1
          fi
          echo "Release ${RELEASE_SHA} confirmed in Sentry."

      - name: Query Sentry issues (new unresolved)
        id: sentry_issues
        env:
          RELEASE_SHA: ${{ steps.release.outputs.sha }}
        run: |
          set -euo pipefail
          curl -sS -f -G "https://sentry.io/api/0/organizations/${SENTRY_ORG}/issues/" \
            -H "Authorization: Bearer ${SENTRY_AUTH_TOKEN}" \
            --data-urlencode "project=${SENTRY_PROJECT}" \
            --data-urlencode "environment=production" \
            --data-urlencode "query=release:${RELEASE_SHA} is:unresolved age:-${OBSERVATION_WINDOW_MINUTES}m" \
            --data-urlencode "sort=new" \
            --data-urlencode "limit=100" \
            -o current-issues.json
          EVENT_COUNT=$(jq '[.[] | .count | tonumber] | add // 0' current-issues.json)
          NEW_ISSUE_COUNT=$(jq 'length' current-issues.json)
          echo "event_count_new_issues=${EVENT_COUNT}" >> "$GITHUB_OUTPUT"
          echo "new_issue_count=${NEW_ISSUE_COUNT}" >> "$GITHUB_OUTPUT"
          echo "Captured ${NEW_ISSUE_COUNT} new unresolved issue(s) with ${EVENT_COUNT} total events."

      - name: Query Sentry sessions (crash-free rate)
        id: sentry_sessions
        env:
          RELEASE_SHA: ${{ steps.release.outputs.sha }}
        run: |
          set -euo pipefail
          curl -sS -f -G "https://sentry.io/api/0/organizations/${SENTRY_ORG}/sessions/" \
            -H "Authorization: Bearer ${SENTRY_AUTH_TOKEN}" \
            --data-urlencode "project=${SENTRY_PROJECT}" \
            --data-urlencode "environment=production" \
            --data-urlencode "field=sum(session)" \
            --data-urlencode "field=crash_free_rate(session)" \
            --data-urlencode "statsPeriod=1h" \
            --data-urlencode "query=release:${RELEASE_SHA}" \
            -o current-sessions.json
          # Default crash_free_rate to 1.0 if no sessions recorded yet (avoid false positive).
          CRASH_FREE=$(jq '[.groups[]? | select((.by // {}) | length == 0) | .totals["crash_free_rate(session)"]] | first // 1' current-sessions.json)
          ERROR_RATE_PCT=$(awk -v cf="${CRASH_FREE}" 'BEGIN { printf "%.4f", (1 - cf) * 100 }')
          echo "error_rate_pct=${ERROR_RATE_PCT}" >> "$GITHUB_OUTPUT"
          echo "Crash-free rate: ${CRASH_FREE} → error rate ${ERROR_RATE_PCT}%"

      - name: Query Sentry transactions (p95 — baseline only)
        id: sentry_transactions
        env:
          RELEASE_SHA: ${{ steps.release.outputs.sha }}
```

<threat_model>
Critical threats surfaced in 43-RESEARCH.md + planner analysis:

**T-43-01 — Secret leakage via logs.** Sentry auth token or GitHub PAT could be printed by accidental `echo ${{ secrets.* }}` or by curl verbose mode (`-v`). Mitigation: all curl calls use `-sS` (silent + show-errors only), no `echo` of `$SENTRY_AUTH_TOKEN`, secrets passed via `env:` block not inline interpolation into shell strings.

**T-43-02 — Unauthorized workflow_dispatch.** Any repo contributor with `actions: write` can trigger the gate with arbitrary release_sha. Mitigation: `release_sha` regex-validated to `^[a-f0-9]{40}$` so injection attempts (e.g., `; rm -rf /`) fail the resolve step before curl. Repo branch protection should also restrict workflow dispatch to maintainers (out-of-scope for this plan — documented as operator action).

**T-43-03 — Preview deploy triggers production gate.** Vercel emits dispatch events for preview + staging deploys too. Mitigation: job `if:` guard requires `github.event.client_payload.environment == 'production'`. Preview deploys will dispatch but the job condition evaluates false, so no Sentry query runs and no artifact is uploaded.

**T-43-04 — Baseline artifact tampering.** An attacker with `actions: write` could upload a crafted baseline artifact to force PASS. Mitigation (partial): the gate reads only the most recent `sentry-baseline-*` artifact — an attacker would need to have uploaded a valid baseline AFTER a real deploy, which implies they already have write access to the repo. No additional signing of baselines; accepted risk.

**T-43-05 — Sentry API downtime blocks deploys.** If Sentry is down, curl fails with non-200, and the gate fails — blocking the deploy. Mitigation: workflow fails (non-zero), but does NOT auto-rollback (D1 decision). Operator sees the failure and can force-re-run the workflow once Sentry is back.

**T-43-06 — Low-traffic false positives.** With < 100 events/day, a single new issue crossing 10 events can trip the gate. Mitigation: D2 threshold defaults set at 10 events + 1.0% rate delta. All three thresholds are repo variables — easy to tune post-launch without code change.

**T-43-07 — First-run deadlock.** Without a prior baseline, the error-rate delta rule cannot evaluate. Mitigation: `first_run: true` placeholder in Task 3 triggers `PASS_WITH_WARNING` exit 0. Baseline is established after first run; second run onward behaves normally.

**T-43-08 — Artifact retention gap.** If two consecutive deploys happen within 30 min (observation window overlap), second deploy's gate could query Sentry before first deploy's artifact is uploaded. Mitigation: concurrency group keyed to release SHA lets two different SHAs run in parallel; same-SHA re-runs do not cancel (so re-runs still produce artifacts). Operator should avoid rapid back-to-back production deploys — documented as deploy practice, not code constraint.
</threat_model>
