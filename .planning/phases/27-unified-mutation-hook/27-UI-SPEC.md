# Phase 27 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase Classification: Data-Layer Only

Phase 27 creates a `useCreateInvitation()` mutation hook and a route constant. It introduces **no new UI components, screens, layouts, or visual changes**. The sole user-facing output is toast feedback messages (success/error) using the existing sonner toast system.

Phase 28 (Consumer Migration) is the UI phase that consumes this hook and introduces visual changes. This contract documents only the toast copywriting and the discriminated result contract that Phase 28 will depend on.

## Copywriting Contract

### Toast Messages (hook `onSuccess` / `onError` callbacks)

| Element | Copy | Source |
|---------|------|--------|
| Success toast (created) | Title: `"Invitation sent"` / Description: `"${tenantName} will receive an email to accept the invitation"` | Matches existing pattern in `use-tenant-invite-mutations.ts` line 31 |
| Success toast (created, no name) | Title: `"Invitation sent"` / Description: `"An invitation email has been sent"` | Fallback when tenant name unavailable |
| Error toast (generic) | Title: `"Failed to send invitation"` / Description: `"Something went wrong. Please try again."` | Via `handleMutationError()` -- existing centralized handler |
| Error toast (duplicate constraint) | Not shown -- hook returns `{ status: 'duplicate' }` result instead of erroring | Decision D-02: duplicate is a valid outcome, not an error |

### Discriminated Result Contract (consumed by Phase 28 UI)

The hook returns a discriminated union. Phase 28's UI-SPEC will define the visual treatment. This contract documents the data shape only.

| Result Status | Data Shape | Phase 28 UI Response (informational) |
|---------------|------------|--------------------------------------|
| `'created'` | `{ status: 'created', invitation: InvitationRecord }` | Success toast (defined above) |
| `'duplicate'` | `{ status: 'duplicate', existing: InvitationRecord }` | Phase 28 defines: "already invited" prompt with resend action |

### Route Constant

| Constant | Value | Verified Against |
|----------|-------|------------------|
| `INVITATION_ACCEPT_PATH` | `'/accept-invite'` | `src/app/(auth)/accept-invite/page.tsx` route file exists |

### No Additional Copywriting Needed

- No empty states (hook produces no UI views)
- No destructive actions (cancel/resend remain in existing hooks per D-06)
- No primary CTA (hook is called programmatically, not via a button it owns)

## Phase 28 Handoff Notes

Phase 28 (Consumer Migration & Dead Code Removal) is the UI-heavy phase. The following design questions are deferred to Phase 28's UI-SPEC:

1. **Duplicate invitation prompt UI**: How to display "This email already has a pending invitation. Resend instead?" -- dialog, inline alert, or toast with action button
2. **Pending invitation list on tenant page**: Row layout, status badges, resend/cancel action buttons, copy-link interaction
3. **Existing user accept flow**: Login redirect UX for registered users clicking invitation links
4. **Empty state for invitation list**: Copy and illustration when no pending invitations exist
