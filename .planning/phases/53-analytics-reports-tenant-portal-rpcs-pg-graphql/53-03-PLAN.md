`tenantPortalQueries.dashboard()` — The NestJS dashboard endpoint aggregated multiple sources. Replace with parallel PostgREST queries in the queryFn:

Get active lease: `supabase.from('leases').select('id, start_date, end_date, rent_amount, lease_status, unit_id, units(unit_number, properties(name, address_line1, city, state, postal_code))').in('lease_tenants.tenant_id', [tenantId])` — or join via `lease_tenants`. `tenantPortalQueries.amountDue()` — This requires late fee calculation logic that is NestJS-specific. Replace with: `supabase.from('rent_payments').select('id, amount_cents, due_date, status, paid_at').eq('lease_id', leaseId).gte('due_date', new Date().toISOString().split('T')[0]).order('due_date').limit(1).single()` — to find next due payment. **maintenance-keys.ts: Fix tenantPortal() stub:**
Replace the TODO stub in `maintenanceQueries.tenantPortal()` with:
```
const supabase = createClient()
const { data: { user } } = await supabase.auth.getUser()
if (!user) return { requests: [], total: 0, open: 0, inProgress: 0, completed: 0 }

// RLS on maintenance_requests uses owner_user_id, but tenants submit requests with their tenant_id
// Filter by tenant_id = auth.uid() directly — tenant owns their own requests
const { data, error, count } = await supabase
  .from('maintenance_requests')
  .select('id, title, description, priority, status, created_at, updated_at, completed_at, unit_id', { count: 'exact' })
  .eq('tenant_id', user.id)
  .order('created_at', { ascending: false })

if (error) handlePostgrestError(error, 'maintenance_requests')

const requests = (data ?? []) as MaintenanceRequest[]
const total = count ?? 0
const open = requests.filter(r => r.status === 'open' || r.status === 'assigned').length
const inProgress = requests.filter(r => r.status === 'in_progress' || r.status === 'needs_reassignment').length
const completed = requests.filter(r => r.status === 'completed').length

return { requests, total, open, inProgress, completed }
```

In that case, leave a comment: `// NOTE(phase-53): RLS on maintenance_requests uses owner_user_id. A separate RLS policy for tenants filtering by tenant_id may be needed.` and return empty data without throwing. The data layer is correct; RLS may need a migration (out of scope for this task — track as a known gap).
