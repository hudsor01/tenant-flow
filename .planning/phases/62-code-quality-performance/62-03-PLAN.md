## Tasks

### T1: Create getCachedUser() utility for non-React contexts
**New file**: `apps/frontend/src/lib/supabase/get-cached-user.ts`

The 86 `getUser()` calls are inside `queryFn`/`mutationFn` — they can't use React hooks. Create a utility that:
1. Reads from the TanStack Query cache (same key as AuthProvider: `['auth', 'session']`)
2. If cache hit with valid session, returns `session.user` (zero network calls)
3. If cache miss, falls back to `supabase.auth.getUser()` (one network call)

```typescript
import { QueryClient } from '@tanstack/react-query'
import { createClient } from './client'
import type { User } from '@supabase/supabase-js'

let queryClientRef: QueryClient | null = null

export function setQueryClientRef(qc: QueryClient) {
  queryClientRef = qc
}

export async function getCachedUser(): Promise<User | null> {
  // Try cache first
  if (queryClientRef) {
    const session = queryClientRef.getQueryData(['auth', 'session'])
    if (session?.user) return session.user
  }
  // Fallback to network
  const supabase = createClient()
  const { data: { user } } = await supabase.auth.getUser()
  return user
}
```

Register the QueryClient ref in AuthProvider or app layout.

### T3: Optimize tenant portal serial lookups
**File**: `apps/frontend/src/hooks/api/use-tenant-portal.ts`

#### T3a: Dashboard query — 3 serial requests to 1
Current: getUser() -> tenants -> leases (with joins)
Fix: Use getCachedUser() + combine tenant + lease into single query:
```typescript
const { data } = await supabase
  .from('tenants')
  .select('id, leases:lease_tenants!inner(lease:leases!inner(id, start_date, end_date, ...))')
  .eq('user_id', user.id)
  .single()
```
Or keep two queries but run tenant lookup only once.

#### T3b: Amount due query — 5 serial requests to 3 parallel
Current: getUser -> tenants -> leases -> stripe_connected_accounts -> rent_due -> rent_payments
Fix: After getting user+tenant+lease (1-2 requests), run stripe_connected_accounts + rent_due in PARALLEL using Promise.all(). The rent_payments check depends on rent_due so stays sequential.

#### T3c: Payments query — 3 serial to 2
Current: getUser -> tenants -> leases -> rent_payments
Fix: getCachedUser() eliminates the getUser round-trip. Combine tenant+lease lookup.

### T5: Batch tenant operations — eliminate N+1
**File**: `apps/frontend/src/hooks/api/use-tenant.ts`

#### T5a: batchUpdate — single query
Replace the `for` loop with a single PostgREST call per unique update payload. If all updates have the same data, use `.in('id', ids)`. If different data per row, group by payload and batch per group.

For the common case (same status update across multiple tenants):
```typescript
const { error } = await supabase
  .from('tenants')
  .update(data)
  .in('id', ids)
```

#### T5b: batchDelete — single query
Replace the `for` loop deleting lease_tenants one at a time with:
```typescript
const { error } = await supabase
  .from('lease_tenants')
  .delete()
  .in('tenant_id', ids)
```
