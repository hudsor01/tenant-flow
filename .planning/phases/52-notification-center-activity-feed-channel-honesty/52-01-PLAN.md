<objective>
Establish the notification write-path primitive that every later v10.0 phase publishes through: the `create_notification` SECURITY DEFINER RPC, a composite inbox index, and an idempotent reconcile of the `notifications` RLS to the correct owner-scoped policy set. Prod currently has only `notifications_select` + `notifications_update` (the richer repo policy set was never applied), so the migration must DROP-IF-EXISTS all legacy policy names and CREATE the canonical set.

Purpose: NOTIF-01. This RPC is the single, un-bypassable system insert point (mirrors `record_lease_signature`). Phases 53/57/62 call it. It must NOT be grantable to `authenticated` (that would let any owner mint notifications for any other owner), must NOT gate on `auth.uid()` (it runs from trigger/cron contexts), and must NOT consult `notification_settings` (D-05: in-app notifications are always created).
Output: one migration applied to prod, regenerated types, and a hardened RLS integration test.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01 | Elevation / Spoofing | create_notification grant | mitigate | `revoke all from public` + `grant execute to service_role` only; NO authenticated grant — asserted by RLS test case (Task 2) |
| T-52-02 | Information disclosure (IDOR) | notifications SELECT | mitigate | `notifications_select` USING `user_id = (select auth.uid())`; dual-client isolation test |
| T-52-03 | Tampering (IDOR) | notifications UPDATE | mitigate | `notifications_update` USING + WITH CHECK owner-scoped; ownerB-cannot-update-ownerA test |
| T-52-04 | Elevation | authenticated INSERT to notifications | mitigate | no authenticated INSERT policy created — clients cannot self-insert |

No npm/pip/cargo installs occur in this plan (zero-new-deps positioning invariant), so the supply-chain (T-52-SC) checkpoint is not triggered.
