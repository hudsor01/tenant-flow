  <action>
Create migration that implements GDPR-compliant account deletion:

**Part A: Add deletion_requested_at column to users**
```sql
alter table public.users add column if not exists deletion_requested_at timestamptz;
```
This enables a 30-day grace period before anonymization executes.

**Part B: Create request_account_deletion() function**
SECURITY DEFINER, SET search_path = public. Validates auth.uid() matches caller. Sets `deletion_requested_at = now()` on the user record. Does NOT immediately anonymize -- the cron job handles that after 30 days.

**Part C: Create anonymize_deleted_user(p_user_id uuid) function**
SECURITY DEFINER, SET search_path = public. This is called by a cron job (not directly by users). It must:

1. **Determine user type** (OWNER vs TENANT) by querying the users table.

2. **For TENANT deletion:**
   - Anonymize tenant record: `update public.tenants set first_name = '[deleted]', last_name = '[deleted]', email = '[deleted]', phone = null where user_id = p_user_id;`
   - Anonymize activity: `update public.activity set description = '[deleted user activity]' where user_id = p_user_id;`
   - Anonymize notifications: `delete from public.notifications where user_id = p_user_id;`
   - Anonymize user_preferences, notification_settings: `delete from ... where user_id = p_user_id;`
   - Keep payment records intact (amounts, dates, lease references) per CONTEXT.md locked decision
   - Remove from lease_tenants: leave records but anonymize any PII fields
   - Mark user record: `update public.users set full_name = '[deleted user]', email = '[deleted]' where id = p_user_id;`

3. **For OWNER deletion:**
   - First CHECK: if owner has active leases, RAISE EXCEPTION 'Cannot delete account with active leases. Please end all leases first.'
   - First CHECK: if owner has pending payments (rent_due with status != 'paid'), RAISE EXCEPTION 'Cannot delete account with pending payments.'
   - Soft-delete all properties: `update public.properties set status = 'inactive' where owner_user_id = p_user_id;`
   - Anonymize activity: same as tenant
   - Clean up notifications, preferences, settings: same as tenant
   - Keep financial records (rent_payments, rent_due) intact for tenant reference
   - Mark user record: same as tenant

4. **Do NOT delete from auth.users** -- that is handled by Supabase Auth separately. This function only anonymizes application data.

**Part D: Create process_account_deletions() cron function**
Runs daily, finds users where `deletion_requested_at < now() - interval '30 days'` and `deletion_requested_at is not null`, calls `anonymize_deleted_user(id)` for each. Uses FOR UPDATE SKIP LOCKED. Handles exceptions per-user (one failure doesn't block others).

**Part E: Schedule the cron job**
```sql
select cron.schedule(
  'process-account-deletions',
  '45 3 * * *',  -- 3:45 AM UTC, after cleanup jobs
  $$select public.process_account_deletions()$$
);
```

**Part F: Create cancel_account_deletion() function**
SECURITY DEFINER. Allows user to cancel within the 30-day grace period:
```sql
update public.users set deletion_requested_at = null where id = (select auth.uid());
```

Header comment: GDPR Article 17 implementation. 30-day grace period. Anonymization preserves financial records per business requirements. Owner deletion blocked if active obligations exist.

IMPORTANT: Before writing the migration, check the actual columns on `users`, `tenants`, `notifications`, `user_preferences`, `notification_settings` tables to ensure column names are correct. Also check if `users.email` actually exists as a separate column (it might be in auth.users only).
