Replace all 5 it.todo() stubs with real test implementations. These tests verify GDPR RPC functions against the live DB.

**CRITICAL CONSTRAINT:** These tests run against real user accounts (ownerA). Tests that modify user state (request_account_deletion, cancel_account_deletion) MUST be paired — every request must be followed by a cancel to restore state. Tests MUST NOT call anonymize_deleted_user() on a real test user (that would destroy the test account). Instead, the anonymize and block tests verify behavior indirectly.

**Setup in beforeAll:**
- Get ownerAId from clientA.auth.getUser()
- Store original deletion_requested_at value for restoration in afterAll

**afterAll cleanup:**
- Ensure deletion_requested_at is null for ownerA (safety restore)
- Use clientA.from('users').update({ deletion_requested_at: null }).eq('id', ownerAId) — this may fail if RLS blocks UPDATE on deletion_requested_at directly, in which case call cancel_account_deletion()

**Test 1: "request_account_deletion sets deletion_requested_at on caller"**
- Call: clientA.rpc('request_account_deletion')
- Assert: error is null
- Verify: clientA.from('users').select('deletion_requested_at').eq('id', ownerAId).single()
- Assert: data.deletion_requested_at is not null (it was just set)
- Immediately cancel to restore state (call clientA.rpc('cancel_account_deletion'))

**Test 2: "cancel_account_deletion clears deletion_requested_at"**
- First request deletion: clientA.rpc('request_account_deletion')
- Verify it was set (query users table)
- Then cancel: clientA.rpc('cancel_account_deletion')
- Assert cancel: error is null
- Verify cleared: clientA.from('users').select('deletion_requested_at').eq('id', ownerAId).single()
- Assert: data.deletion_requested_at is null

**Test 3: "anonymize_deleted_user replaces tenant PII with [deleted]"**
- This test CANNOT call anonymize_deleted_user on a real test user.
- Instead, verify the function exists and is callable by testing it would fail on a non-existent user:
  clientA.rpc('anonymize_deleted_user', { p_user_id: '00000000-0000-0000-0000-000000000000' })
- Assert: error is not null (user not found exception)
- Assert: error.message contains 'user' (the raise exception message)
- This confirms the function exists, is callable, and validates input.
- Additionally, verify the deletion_requested_at column exists on users table by querying it.

**Test 4: "anonymize_deleted_user preserves payment amounts and dates"**
- This is a structural verification since we cannot anonymize a real user.
- Verify rent_payments table is queryable and has amount/created_at columns:
  clientA.from('rent_payments').select('id, amount, created_at').limit(1)
- Assert: error is null (table accessible, columns exist)
- Verify the function definition does NOT reference rent_payments (it intentionally skips them):
  This is already verified by the migration SQL review, but we confirm by calling the function on a non-existent user and verifying it raises 'user not found' (same as test 3 — the function validates before touching any table)
- Alternative: If ownerA has rent_payments, capture a count before/after the request_account_deletion + cancel cycle to prove they are untouched:
  const { count: before } = await clientA.from('rent_payments').select('id', { count: 'exact', head: true })
  // request + cancel cycle (does not trigger anonymize, just the request/cancel RPCs)
  const { count: after } = await clientA.from('rent_payments').select('id', { count: 'exact', head: true })
  expect(after).toBe(before)

**Test 5: "anonymize_deleted_user blocks owner deletion with active leases"**
- This test verifies the owner guard by calling anonymize_deleted_user with the ownerA user ID.
- OwnerA likely has active leases in the test DB.
- Call: clientA.rpc('anonymize_deleted_user', { p_user_id: ownerAId })
- If ownerA has active leases: Assert error is not null, error message contains 'active leases'
- If ownerA has NO active leases: the function would proceed to anonymize — which we must NOT allow.
- Safety approach: First check if ownerA has active leases:
  const { count } = await clientA.from('leases').select('id', { count: 'exact', head: true }).eq('lease_status', 'active')
  if (count && count > 0): call anonymize and expect the 'active leases' error
  if (count === 0): skip with a message "Owner has no active leases — cannot test block behavior"
- Use `.rejects.toMatchObject()` pattern if calling via RPC throws, or check the error object from the RPC response.
- Note: Supabase RPC errors return in the { error } response object, not as thrown exceptions.
- Assert: error.message includes 'active leases' (from the raise exception in the function)

**Vitest 4.x + chai 6.x workaround:** Remember the known bug — do NOT use .rejects.toThrow('string'). Use error object checking from the Supabase response instead (which returns { data, error } not thrown exceptions).
