<threat_model>
Critical threats surfaced in 44-RESEARCH.md + planner analysis. STRIDE categories mapped per project security_enforcement rules.

**T-44-01 — Svix signature bypass (Spoofing).** Category: Spoofing. Component: `supabase/functions/resend-webhook/index.ts`. An attacker forges a Resend webhook POST to fabricate deliverability events (e.g., claim 100% delivery for a bounced campaign). Disposition: mitigate. Mitigation: Svix HMAC-SHA256 verification BEFORE any DB write, with constant-time compare via `crypto.subtle.timingSafeEqual` and XOR fallback; 5-minute timestamp tolerance blocks replay outside the window; secret is `whsec_<base64>` stored in Supabase Edge Function secrets (not in code). Reject 400 on any mismatch. Reference: RESEARCH.md Pattern 1 lines 193-260 + docuseal-webhook template.

**T-44-02 — Replay attack within freshness window (Tampering).** Category: Tampering. Component: resend-webhook Edge Function. An attacker captures a valid signed payload and re-POSTs within 5 minutes. Disposition: mitigate. Mitigation: Pitfall 2 defense — DB upsert uses `onConflict: 'message_id,event_type'` so replays are idempotent (upsert does nothing on exact duplicate). Within-window replay cannot fabricate events because the signed `{id, timestamp, body}` tuple is bound — attacker cannot alter the payload without invalidating the signature.

**T-44-03 — message_id collision fabrication (Tampering).** Category: Tampering. Component: `email_deliverability` table. An attacker (post-Svix bypass) submits crafted message_id values to overwrite legitimate entries. Disposition: accept (conditional on T-44-01 mitigation holding). Mitigation: UNIQUE composite `(message_id, event_type)` prevents overwrite — second insert with same composite key is idempotent upsert. Only the `raw_payload` can be rewritten, and rewriting is symmetric (replay of the same event has no semantic effect on aggregates). No additional defense needed beyond Svix signature.

**T-44-04 — Admin RPC bypass (Elevation of Privilege).** Category: Elevation of Privilege. Component: `public.get_deliverability_stats`. A non-admin authenticated user calls the RPC to read aggregate deliverability (including recipient email patterns + bounce rates — marketing-sensitive). Disposition: mitigate. Mitigation: `if not is_admin() then raise exception 'Unauthorized'` as first statement after p_days range check (cheap validation runs first). Task 6's RLS integration test proves non-admin is rejected. `grant execute to authenticated` is safe because `is_admin()` guard enforces at runtime.

**T-44-05 — Insecure error leakage (Information Disclosure).** Category: Information Disclosure. Component: resend-webhook Edge Function. Raw PostgREST / Supabase error messages could leak DB schema, constraint names, or row contents to an attacker probing the webhook with malformed payloads. Disposition: mitigate. Mitigation: all client-facing errors go through `errorResponse()` from `_shared/errors.ts` (returns generic `{error: 'An error occurred'}`). Real errors logged server-side via `console.error` + Sentry. No `err.message` ever reaches the Response body. Applies to signature failure, body parse failure, validation failure, DB upsert failure.

**T-44-06 — Cron job failure silent degradation (Denial of Service).** Category: Denial of Service. Component: `cleanup_old_email_deliverability` cron. If the cleanup cron fails silently (e.g., archive table out of disk), `email_deliverability` grows unbounded and eventually blocks ingest + RPC. Disposition: mitigate. Mitigation: existing `check_cron_health` cron (CLAUDE.md cron jobs table) runs hourly, logs failures to `user_errors` for Sentry pickup. The new `cleanup-email-deliverability` job is automatically monitored by `check_cron_health` because it queries `cron.job_run_details` for all jobs. No new monitoring code needed.

**T-44-07 — Env secret leakage (Information Disclosure).** Category: Information Disclosure. Component: resend-webhook Edge Function env. `RESEND_WEBHOOK_SECRET`, `SUPABASE_SERVICE_ROLE_KEY` could leak via accidental `console.log(env)` or error stack traces. Disposition: mitigate. Mitigation: `validateEnv` returns a typed object but never logs values; `console.error(err)` is used only for err objects which do not include env; error responses go through `errorResponse()` which never echoes env. Secrets are set via `supabase secrets set` not `.env` files committed to git. No code change needed — enforced by convention + absence of `console.log(env)` in the written code.

**T-44-08 — Tag injection in template_tag (Tampering).** Category: Tampering. Component: `extractTemplateTag` in Edge Function. An attacker-controlled Resend tag value could contain SQL/HTML metacharacters. Disposition: accept. Mitigation: `template_tag` is stored as `text` via parameterized insert (Supabase client uses prepared statements internally — no SQL injection surface). The RPC `GROUP BY template_tag` does not render it as HTML or execute it as SQL. Even if an attacker controls Resend tags (they cannot — sender controls tags), the worst case is a weird-looking template tag in the admin dashboard, which is a UX nit not a security breach. Admin UI in Plan 03 will render via React (escape by default).

**T-44-09 — Timestamp freshness bypass (Replay attack).** Category: Tampering. Component: `verifySvixSignature`. Attacker backdates `svix-timestamp` to match a captured-signature-later scenario, OR clock skew between Resend and Supabase causes legitimate payloads to be rejected. Disposition: mitigate. Mitigation: 5-minute tolerance is the Svix-documented standard — tight enough to block replay, loose enough to absorb NTP jitter. `Math.abs(nowSec - tsSec) > 300` check blocks both backdating and future-dating. The check runs BEFORE HMAC verification so an out-of-window signature failure surfaces as 400 without expensive HMAC work.
</threat_model>
