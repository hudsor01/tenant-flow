  } catch { return false }
}

**Step D: Verify login redirect works**

The login page at `src/app/(auth)/login/page.tsx` already:
1. Reads `searchParams?.get('redirect')` (line 93)
2. Validates via `isValidRedirect()` (line 101)
3. Uses the redirect as destination after successful login (line 102)

The URL will be: `/login?redirect=%2Faccept-invite%3Fcode%3DXXX`

When login reads `searchParams.get('redirect')`, it gets the decoded value `/accept-invite?code=XXX` (browsers decode `%2F` -> `/`, `%3F` -> `?`, `%3D` -> `=`). Actually, `URLSearchParams.get()` returns the raw decoded value. So `isValidRedirect('/accept-invite?code=XXX')` checks:
1. Starts with `/` -- YES
2. Does not start with `//` -- YES
3. `new URL('/accept-invite?code=XXX', window.location.origin)` -- valid URL
4. Hostname matches -- YES

The redirect should work correctly. No changes needed to the login page (per D-10, confirming existing support is sufficient).

**IMPORTANT -- URL encoding:**
The redirect URL MUST be encoded with `encodeURIComponent()`. Without encoding, `/login?redirect=/accept-invite?code=XXX` would cause `searchParams.get('redirect')` to return only `/accept-invite` (the `?code=XXX` would become part of the login page's own query params, not the redirect value).
