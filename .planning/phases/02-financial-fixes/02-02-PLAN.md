Modify `stripe-webhooks/index.ts` with these changes:

**PAY-17: Stripe client initialization** — Add `apiVersion: '2026-02-25.clover'` to the Stripe constructor. If `constructEventAsync` API changed in v20, adapt accordingly (may now be `stripe.webhooks.constructEventAsync` or similar).

**PAY-15: Idempotency with status tracking** — Replace the current idempotency pattern:
- On insert to `stripe_webhook_events`, include `status: 'processing'` (new column from 02-01 migration).
- On duplicate key error (23505), check the existing record's `status`:
  - If `'failed'`: update to `'processing'` and continue (allow retry).
  - If `'succeeded'` or `'processing'`: return 200 with `{ received: true, duplicate: true }`.
- After successful processing: update status to `'succeeded'`.
- On processing error: update status to `'failed'` and `error_message` to `err.message`. Do NOT delete the record. Return 500.
- Use the exact pattern from RESEARCH.md Pattern 3.

**PAY-10: Metadata validation** — In the `payment_intent.succeeded` handler, before processing:
- Extract `tenant_id`, `lease_id`, `rent_due_id`, `unit_id` from `pi.metadata`.
- If `tenant_id` or `lease_id` is missing or empty string, capture a structured error to Sentry (`Sentry.captureException` or `Sentry.captureMessage`) with the event ID, metadata fields present, and PaymentIntent ID as context, then return early (do not throw — this is a data issue, not transient). Per Decision #15, Sentry is mandatory for metadata validation failures.
- Remove any `?? ''` or `|| ''` fallbacks on metadata fields.
- Note: If Sentry is not already imported in this Edge Function, use `console.error` with structured JSON including `{ sentry: true, event_id, pi_id, metadata }` so the Sentry Edge Function log drain can pick it up. Document the Sentry integration approach in the SUMMARY.

**PAY-11: onboarding_completed_at preservation** — In the `account.updated` handler:
- Before setting `onboarding_completed_at`, check if it is already set.
- Only set `onboarding_completed_at = new Date().toISOString()` when: (a) `charges_enabled` is true in the account event, AND (b) the current DB value is null.
- Query the users table first: `SELECT onboarding_completed_at FROM users WHERE stripe_account_id = account.id`. If already set, skip the update.

**PAY-18: Fee breakdown in owner receipt** — In the `payment_intent.succeeded` handler, when sending the owner notification email:
- Extract fee information from the PaymentIntent's `charges` or use `latest_charge` to get fee details.
- Retrieve the charge: `const charge = await stripe.charges.retrieve(pi.latest_charge as string)`.
- Calculate: `grossAmount = pi.amount / 100` (convert from cents), `stripeFee = charge.balance_transaction ? (await stripe.balanceTransactions.retrieve(charge.balance_transaction as string)).fee / 100 : 0`, `applicationFee = pi.application_fee_amount ? pi.application_fee_amount / 100 : 0`, `netAmount = grossAmount - stripeFee - applicationFee`.
- Pass these amounts to the owner notification email template.
- Note: This is informational. If balance_transaction retrieval fails, log and continue without fee breakdown rather than failing the webhook.

**PAY-02: Use record_rent_payment RPC** — Replace the direct `rent_payments` INSERT in `payment_intent.succeeded` with a call to the `record_rent_payment` RPC (created in 02-01). This atomically records the payment AND updates `rent_due.status`. Call via `supabase.rpc('record_rent_payment', { ... })`.

**PAY-09: Add invoice.payment_failed handler** — Add a new case in the processEvent switch:
```typescript
case 'invoice.payment_failed': {
  const invoice = event.data.object
  const customerId = invoice.customer as string

  // Look up owner by stripe_customer_id
  const { data: owner } = await supabase
    .from('users')
    .select('id, email, first_name, stripe_customer_id')
    .eq('stripe_customer_id', customerId)
    .single()

  if (!owner) {
    console.error(`[WEBHOOK] invoice.payment_failed: no user found for customer ${customerId}`)
    break
  }

  // Update subscription status tracking if needed
  // The stripe.subscriptions table is synced separately, but we can
  // update the user record to flag the failure for immediate UI response
  const { error: updateError } = await supabase
    .from('users')
    .update({ subscription_status: 'past_due' })
    .eq('id', owner.id)

  if (updateError) {
    console.error(`[WEBHOOK] Failed to update subscription status for user ${owner.id}:`, updateError)
  }

  // Send owner notification email about subscription payment failure
  const { sendEmail } = await import('../_shared/resend.ts')
  await sendEmail({
    to: [owner.email],
    subject: 'Action Required: Subscription Payment Failed',
    html: `<p>Hi ${owner.first_name || 'there'},</p>
      <p>Your TenantFlow subscription payment failed. Please update your payment method to avoid service interruption.</p>
      <p>You have a 7-day grace period before premium features are restricted.</p>
      <p><a href="${Deno.env.get('FRONTEND_URL') || 'https://app.tenantflow.app'}/owner/billing">Update Payment Method</a></p>`,
    tags: [{ name: 'category', value: 'subscription_failure' }]
  })

  console.log(`[WEBHOOK] invoice.payment_failed processed for user ${owner.id}`)
  break
}
```
Note: If `users` table does not have a `subscription_status` column, skip the direct status update — the `stripe.subscriptions` table (synced by Stripe extension) is the source of truth. The email notification is the primary action here. Check for the column existence before attempting the update or use a try/catch.
