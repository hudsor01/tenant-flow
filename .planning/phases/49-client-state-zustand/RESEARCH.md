## STATE-01 — Stranded pagination page + stale selection on leases list
- **Finding:** src/app/(owner)/leases/page.tsx:155 (medium) — `currentPage` persists in the module-level leases store and is never clamped to the recomputed `totalPages`, so a shrunken dataset strands the user on an empty page with the pagination footer hidden (`totalPages > 1` guard at leases-table.tsx:221) and no recovery control; `selectedRows` likewise retains ids of leases deleted elsewhere.
- **Root cause:** Derived pagination/selection state in a module-singleton store is never reconciled against the server dataset it indexes into. `setCurrentPage` is only reset by `setSearchQuery`/`setStatusFilter` (leases-store.ts:116-122), and page.tsx:154-158 slices with the raw stored `currentPage`; nothing intersects `selectedRows` with the fetched id set.
- **Fix:**
  1. In `src/app/(owner)/leases/page.tsx`, derive a clamped page and use it everywhere: `const effectivePage = Math.min(currentPage, Math.max(1, totalPages))` — use `effectivePage` for the `paginatedLeases` slice and as the `currentPage` prop to `LeasesTable`; add a small effect that writes the clamp back to the store when they diverge (`useEffect(() => { if (effectivePage !== currentPage) setCurrentPage(effectivePage) }, [effectivePage, currentPage, setCurrentPage])`) so the stale large value cannot resurrect later when the dataset grows again.
  2. Class-wide selection reconciliation (shared with STATE-05/STATE-12): add a `pruneSelection(validIds: string[])` action to `src/stores/leases-store.ts` that intersects `selectedRows` with `validIds` and returns the SAME state object when nothing was removed (no-op guard against render loops). In page.tsx add `useEffect(() => { pruneSelection(leases.map(l => l.id)) }, [leases, pruneSelection])` — pruning against the full fetched set (not the filtered set), so filters never wipe a selection but deletions anywhere (detail page, future bulk) do.
- **Why it fixes it:** The verifier confirmed the stranded state is `currentPage=2` with `totalPages=1` and zero rows + no footer. Deriving `effectivePage` makes an out-of-range page mathematically impossible at render time (slice always lands on a real page), and the write-back keeps the store consistent so the footer/counter render from a real page. `pruneSelection` on the fetched id set removes the exact phantom-id retention the verifier confirmed ("`selectedRows` is likewise never pruned").
- **Risks / interactions:** Same files as STATE-03/STATE-08 — implement the three findings as one coordinated edit to `leases-store.ts` + `page.tsx` + `leases-dialogs.tsx`. Prune only runs with loaded data (page early-returns on `isLoading`), so transient empty fetches cannot wipe selection. The clamp effect must clamp to `max(1, totalPages)` because `totalPages` is 0 when the filtered set is empty. Phase 39 (DATA) may have touched this page's money display; build against post-Phase-48 main.
- **Files touched:** src/stores/leases-store.ts, src/app/(owner)/leases/page.tsx
- **Decision:** Clamp+prune (chosen) vs reset-on-mount of the page slice. Reset-on-mount also fixes the audit scenario but throws away legitimate in-session state (page position when nothing changed, selections across a detail-view round-trip) and does NOT fix shrinkage while mounted (`refetchOnWindowFocus: true` can shrink the dataset without a remount). Clamp+prune is the root-cause reconciliation.

## STATE-02 — Sign-out never resets Zustand stores (cross-user state leak)
- **Finding:** src/hooks/api/use-auth.ts:159 (medium) — `clearAuthData()` clears only the TanStack Query cache and `REACT_QUERY_OFFLINE_CACHE`; every module-singleton store (leases, properties, tenants, dashboard, dashboard-presets, navigation) survives sign-out/sign-in SPA navigation, leaking user A's search terms, filters, selections, an in-memory `Lease` entity, and the non-user-scoped `tenantflow-dashboard-presets` localStorage key to user B.
- **Root cause:** Module-level Zustand stores live for the lifetime of the JS realm and every sign-out path is a reload-free SPA transition (verified: app-shell.tsx:296 mutation, signout page `router.push("/login")`, mobile-nav server action → client transition, login `router.push(destination)`); no code path calls the stores' `reset()` actions (zero callers confirmed) and the persisted presets slice is neither user-scoped nor removed.
- **Fix:** Add `src/stores/reset-all-stores.ts` exporting a plain `resetAllStores(): void` that resets every module-singleton store to its initial state:
  - `useLeasesStore.getState().reset()`, `usePropertiesStore.getState().reset()`, `useTenantsStore.getState().reset()`
  - `useDashboardStore.setState({ viewMode: "table" })` (its documented default)
  - `useNavigationStore.setState({ isMobileMenuOpen: false })` (post-STATE-09 shape)
  - `useErrorBoundaryStore.getState().clearError()`
  - `useDashboardPresetsStore.setState({ presets: {}, columnVisibility: {} })` followed by `useDashboardPresetsStore.persist.clearStorage()` (removes the `tenantflow-dashboard-presets` localStorage key)
  Call it from BOTH chokepoints: (1) inside `clearAuthData()` in `src/hooks/api/use-auth.ts` (deterministic on the sign-out mutation path), and (2) inside the `SIGNED_OUT` branch of `onAuthStateChange` in `src/providers/auth-provider.tsx:61-64` (covers the mobile-nav server-action path, session expiry, and cross-tab sign-out, where `clearAuthData` never runs). The function is idempotent so double invocation is harmless. Note: the loading and toast stores need no entry because STATE-04/STATE-06 delete them; the preferences store is provider-scoped and holds non-sensitive device prefs (theme/density/view) shared like the theme cookie — deliberately excluded.
- **Why it fixes it:** The verifier's evidence is that `reset()` has zero callers and all sign-out/sign-in paths are SPA navigations. Wiring `resetAllStores()` into the two sign-out chokepoints guarantees every store is back to `initialState` before user B can render any owner page, and clearing the persist storage removes the cross-user localStorage leak the finding names explicitly.
- **Risks / interactions:** Import graph: `reset-all-stores.ts` imports all stores — it is a function module, not a barrel re-export (rule 2 not violated). Phase 37 (AUTH) may have modified `clearAuthData`/auth-provider; apply on top of its final shape. Phase 42 (DASH) may reshape the dashboard stores; the reset entries must match their post-42 state shape. Clearing presets on sign-out means a returning user re-creates saved presets (see Decision).
- **Files touched:** src/stores/reset-all-stores.ts (new), src/hooks/api/use-auth.ts, src/providers/auth-provider.tsx
- **Decision:** Clear-presets-on-sign-out (chosen: 2-line, privacy-correct, least blast radius) vs user-scoping the persist key (`tenantflow-dashboard-presets:<userId>` via `persist.setOptions({ name })` + `rehydrate()` after auth resolves, preserving each user's presets across sessions on a shared device). User-scoping is the richer UX but requires wiring auth identity into store hydration timing (flash-of-wrong-presets risk) and a migration for the existing unscoped key; if product later wants durable presets, promote them to a DB table instead.

## STATE-03 — Stale server `Lease` entity + dialog flags persist across unmounts
- **Finding:** src/stores/leases-store.ts:51 (medium) — `selectedLease` caches a full server `Lease` row in a global store and `showRenewDialog`/`showTerminateDialog` survive unmount, so back/forward navigation auto-reopens the renew/terminate dialog rendering a stale snapshot (`RenewLeaseDialog` reads `lease.end_date`/`lease.rent_amount` from the stored copy).
- **Root cause:** Server state (a `Lease` entity) is duplicated into Zustand — violating the "TanStack Query owns server state" architecture rule — and no unmount/reset hook exists, so both the snapshot and the open flags outlive the page (`closeAllDialogs`/`reset` have zero callers).
- **Fix:**
  1. In `src/stores/leases-store.ts`: replace `selectedLease: Lease | null` with `selectedLeaseId: string | null`; change `openRenewDialog`/`openTerminateDialog` (and the close actions / `closeAllDialogs` / `initialState`) to operate on the id. Drop the now-unused `import type { Lease }`.
  2. In `src/app/(owner)/leases/page.tsx`: `handleRenew`/`handleTerminate` pass `lease.id`; derive the entity fresh from the query-backed list on every render: `const selectedLease = leases.find(l => l.id === selectedLeaseId)?.original ?? null` (`LeaseDisplay.original: Lease`, lease-utils.ts:44) and pass that to `LeasesDialogs`. Add unmount cleanup: `useEffect(() => closeAllDialogs, [closeAllDialogs])` — giving `closeAllDialogs` its first caller and killing the back/forward auto-reopen.
  3. `src/components/leases/dialogs/leases-dialogs.tsx` keeps its `if (!selectedLease) return null` guard — it now doubles as the "lease disappeared from cache" auto-dismiss.
- **Why it fixes it:** The verifier confirmed the two mechanisms: (a) flags + snapshot survive SPA navigation and are fed straight back into `LeasesDialogs`, and (b) the snapshot is genuinely stale because refetches update only the query cache. Deriving from `leases` (the `useLeaseList` result) makes every dialog render read the live cache — `refetchOnWindowFocus` updates flow into an open dialog's props — and the unmount `closeAllDialogs` guarantees no dialog state survives leaving the page, eliminating the auto-reopen.
- **Risks / interactions:** Same three files as STATE-01/STATE-08 — one coordinated edit. If the lease vanishes from the cache while a dialog is open, the dialog unmounts without `onOpenChange` firing; flags stay `true` with a dangling id, which renders nothing and is wiped by the unmount cleanup — acceptable, note in tests. `RenewLeaseDialog`'s internal `useState` defaults are computed from the `lease` prop on first mount only; because `LeasesDialogs` unmounts when `selectedLease` is null and dialogs are keyed by open/close lifecycle, a fresh open always mounts with fresh data. Phase 38 (FORM) may have reworked renew/terminate dialog internals — adapt prop plumbing to their final shape.
- **Files touched:** src/stores/leases-store.ts, src/app/(owner)/leases/page.tsx, src/components/leases/dialogs/leases-dialogs.tsx

## STATE-05 — Phantom `selectedRows` can resurrect a soft-deleted property via bulk edit
- **Finding:** src/stores/properties-store.ts:34 (medium) — single delete (properties/page.tsx:156-161) soft-deletes to `status: 'inactive'` without pruning the store's `selectedRows`; a subsequent Bulk Edit maps over the stale set and `update({status:'active'}).eq('id', id)` un-deletes the property; the PropertyActionBar also shows a phantom selection count.
- **Root cause:** Same class as STATE-01/STATE-12 — selection state in a module-singleton store is never reconciled against the fetched dataset, and the bulk mutation trusts the raw stored set instead of intersecting with what is actually listed.
- **Fix:** Two layers (both needed — the effect alone leaves a refetch race window):
  1. Class-wide reconciliation: add `pruneSelection(validIds: string[])` to `src/stores/properties-store.ts` (same no-op-guarded intersection as leases/tenants) and call it from `src/components/properties/properties.tsx` via `useEffect(() => { pruneSelection(properties.map(p => p.id)) }, [properties, pruneSelection])` — `properties` already excludes inactive rows (`.neq('status','inactive')` in property-keys list query), so any soft-deleted id leaves the selection as soon as the invalidated query refetches.
  2. Mutation-time guard: in `handleBulkEditSubmit` (properties.tsx:89-109) and in `useBulkHandlers.handleBulkDelete` / `handleBulkEditOpen` (properties-filters.tsx:100-131), compute ids as the intersection of `selectedRows` with the `properties` prop (`const ids = Array.from(selectedRows).filter(id => properties.some(p => p.id === id))`; bail if empty). This closes the delete→refetch race where a phantom id is still selected when the user submits.
- **Why it fixes it:** The verifier traced the resurrection path end-to-end (stale set → `propertyMutations.update()` unguarded `.eq('id', id)`). Pruning removes the phantom id as soon as the list reflects the delete (fixing the phantom action-bar count), and the mutation-time intersection makes it impossible for a bulk write to target an id that is not currently rendered — even mid-race — so `status:'active'` can never be applied to the just-deleted property.
- **Risks / interactions:** Intersecting against the `properties` prop (all fetched properties) rather than `filteredProperties` is deliberate — changing a filter must not silently shrink what a bulk action targets while rows are merely hidden; only truly absent (deleted/inactive) rows are dropped. The action-bar count uses `selectedRows.size`, which stays consistent because pruning fires on the same render cycle as the refetched list. Phase 41/46 may have restyled these components; logic hooks are unchanged.
- **Files touched:** src/stores/properties-store.ts, src/components/properties/properties.tsx, src/components/properties/properties-filters.tsx
