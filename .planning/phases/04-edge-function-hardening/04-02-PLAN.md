  <action>
1. **Create `_shared/rate-limit.ts`** per CONTEXT.md locked decision and RESEARCH.md pattern:
   - Export `rateLimit(req: Request, options: { maxRequests: number; windowMs: number; prefix?: string }): Promise<Response | null>`
   - Uses `@upstash/ratelimit` with `Ratelimit.slidingWindow` algorithm (deps added to deno.json in Plan 01)
   - IP extraction: `x-forwarded-for` first segment, then `cf-connecting-ip`, fallback `'unknown'`
   - Identifier: `{prefix}:{ip}`
   - On rate limit hit: return 429 with `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers, log as warning (structured JSON with IP + endpoint)
   - On not rate limited: return `null` (caller proceeds)
   - On rate limiter error (Upstash down): `console.error`, return `null` (fail-open for availability)
   - Lazy-init the Ratelimit instance (module-level `let` cached)
   - Reads `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` from Deno.env; throws if missing when first called

2. **Wire rate limiting into 3 Edge Functions** -- add right after CORS preflight check, before any business logic:
   - `tenant-invitation-accept/index.ts`: `rateLimit(req, { maxRequests: 10, windowMs: 60_000, prefix: 'invite-accept' })`
   - `tenant-invitation-validate/index.ts`: `rateLimit(req, { maxRequests: 10, windowMs: 60_000, prefix: 'invite-validate' })`
   - `stripe-checkout-session/index.ts`: `rateLimit(req, { maxRequests: 10, windowMs: 60_000, prefix: 'checkout-session' })`
   - Pattern: `const rateLimited = await rateLimit(req, { ... }); if (rateLimited) return rateLimited;`

3. **Cap stripe-connect limit parameter** (EDGE-08): In `stripe-connect/index.ts`, find where `body.limit` is used and cap it: `const limit = Math.min(Number(body.limit) || 10, 100)`. Use `limit` variable for all subsequent Stripe API calls in that function.

4. **Rate limit Sentry tunnel in proxy.ts** (EDGE-11): Since `/monitoring` is a Next.js tunnelRoute (handled by `@sentry/nextjs`), rate limit in the root middleware (`proxy.ts`):
   - Add an in-memory `Map<string, { count: number; resetAt: number }>` at module level (Next.js middleware is a persistent process, unlike Edge Functions)
   - Before `updateSession`, check if `pathname === '/monitoring'` or `pathname.startsWith('/monitoring')`
   - If so, extract IP from `x-forwarded-for` or `req.headers.get('x-real-ip')` or `'unknown'`
   - Apply sliding window: 60 req/min per IP. If exceeded, return `NextResponse.json({ error: 'Too many requests' }, { status: 429 })` with `Retry-After` header
   - Clean up expired entries periodically (every 100 requests, remove entries where `resetAt < now`)

5. **Add env validation + error sanitization** (EDGE-01, EDGE-07) to the 4 functions this plan already modifies -- since we have the files open, apply the full hardening:
   - **tenant-invitation-accept**: required=[SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, SUPABASE_ANON_KEY], optional=[]. Error leaks at lines 100, 132, 143 (`tenantError.message`, `updateError.message`, `err.message`) -- replace with `errorResponse(req, status, error, { action: '...' })`
   - **tenant-invitation-validate**: required=[SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY], optional=[]. Error leak at line 98 (`err.message`)
   - **stripe-checkout-session**: required=[STRIPE_SECRET_KEY], optional=[FRONTEND_URL]. Error leak at line 56 (`err.message`)
   - **stripe-connect**: required=[SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, STRIPE_SECRET_KEY], optional=[FRONTEND_URL]. Error leaks at lines 54, 274 (`dbError.message`, `err.message`)

   For each: import `validateEnv` from `../_shared/env.ts` and `errorResponse` from `../_shared/errors.ts`. Call `validateEnv` inside handler after CORS preflight, wrap in try/catch with errorResponse on failure. Replace all error message leaks with errorResponse calls.
