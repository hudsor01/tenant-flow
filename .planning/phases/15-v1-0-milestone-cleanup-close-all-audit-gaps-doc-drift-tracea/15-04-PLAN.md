<objective>
Eliminate the ~15 unrelated flaky failures observed under the full 105k-test parallel run
(documented in `.planning/phases/12-seo-metadata-schema-content-cleanup/deferred-items.md`)
by tuning the `unit` project's pool / poolOptions in `vitest.config.ts`.

Root cause hypothesis (per deferred-items.md): vitest worker-pool resource exhaustion during
the full 105k-test parallel run produces userEvent / hook timeouts that don't reproduce in
isolation (`property-form.test.tsx` passes 20/20 in 1.18s alone vs lost worker threads
in the full run; 176s wall-clock).

Mandatory diagnostic-first sequence per D-09 / D-10 / D-11:
1. Diagnostic: 3 consecutive `vitest --reporter=json` runs to identify deterministic-fail
   vs flake-only failures.
2. Tune: adjust `pool` / `poolOptions.threads.maxThreads` / `minThreads`. Start with
   `pool: 'threads'` + `maxThreads: 4` + `minThreads: 1` per PATTERNS.md.
3. Verify: 3 consecutive zero-flake runs under new config.
4. Graceful fallback: if tuning fails after reasonable iteration, document diagnostic +
   ship deferral with tracked issue (do NOT ship config changes that don't actually fix
   the symptom — see D-11).

Project-split escape hatch (D-10 step 3) is deferred to a follow-up phase per Deferred Ideas
in CONTEXT.md — Plan 15-04 sticks with pool-tuning only.

Per CLAUDE.md Zero Tolerance Rules: no `any` types in vitest.config.ts edits, no
`as unknown as`. Vitest 4.x `poolOptions.threads` schema is typed.

Purpose: Get the full unit suite deterministic so the v1.0 milestone can archive without a
known flakiness drag.

Output: Tuned `vitest.config.ts` + diagnostic data in `15-04-SUMMARY.md`.
</objective>
