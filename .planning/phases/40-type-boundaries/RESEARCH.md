## TYPE-01 — property detail select drops acquisition columns, edit wipes them
- **Finding:** src/hooks/api/query-keys/property-keys.ts:28 (high) — `PROPERTY_SELECT_COLUMNS` omits `acquisition_cost`/`acquisition_date`, so editing any property nulls both columns in the DB.
- **Root cause:** `PROPERTY_SELECT_COLUMNS` (line 27-28) lists an explicit column subset that leaves out `acquisition_cost` and `acquisition_date`, but both `detail()` (line 122) and `list()` (line 76) cast the row to the full `Property` (= `Tables<"properties">`, core.ts:120), which types those fields as `number|null`/`string|null`. The type lies: at runtime they are `undefined`. Both edit routes ([id]/edit/page.tsx:23 and @modal/(.)edit/[id]/page.tsx:33) hydrate `PropertyForm` from `propertyQueries.detail(id)`. The form defaults collapse the runtime-`undefined` fields to `null`/`""` (property-form.client.tsx:111-115), and `handleEditSubmit` then sends **explicit** `{ acquisition_cost: null, acquisition_date: null }` (lines 218-223) because `value.acquisition_cost === null` / `value.acquisition_date === ""`. `omitUndefined` keeps null (strips only undefined), so `propertyMutations.update` writes null into both columns on every edit. supabase.ts:1301-1302 confirms both are real `properties` columns.
- **Fix:** Add `acquisition_cost, acquisition_date` to `PROPERTY_SELECT_COLUMNS` (line 27-28). One-line change; the constant is shared by `list()` and `detail()`, so the detail/edit hydration path now receives the real stored values. The form defaults then populate with the true values and `handleEditSubmit` sends the real `acquisition_cost`/`acquisition_date` (or null only when the user genuinely clears them). No form change needed — the form logic is correct once fed real data.
- **Why it fixes it:** The verifier chain is: unselected column → runtime `undefined` → form default `null`/`""` → explicit `null` submit → `omitUndefined` preserves null → column wiped. Selecting the columns breaks the chain at step 1, so the form is seeded with the stored values and preserves them on save.
- **Risks / interactions:** Additive only — two extra columns on a select cannot break the list cards (they don't read acquisition fields) or `withUnits()` (already `select("*, units(*)")`). Phase 38 FORM executes before 40 and may touch property-form.client.tsx; TYPE-01's fix is entirely in the query layer, orthogonal to any form-field change, so no ordering dependency. No new type needed.
- **Files touched:** src/hooks/api/query-keys/property-keys.ts

This is the same class as the DATA-01 mis-wire already resolved for occupancy/lease in this file — the RPC shape is not the page-data shape and `jsonObject` doesn't validate keys.
- **Fix:** Replace the blind cast with a boundary mapper `mapFinancialOverview(raw): FinancialAnalyticsPageData`, added to `src/hooks/api/query-keys/analytics-mappers.ts` (the established home next to `mapOccupancyAnalytics`), never-throws, degrades to an empty shape on non-object input. Two parts:

Leave `breakdown.revenue`, `billingInsights`, `invoiceSummary`, `leaseAnalytics` at their empty shapes — no source exists post rent-payment demolish (consistent with the DATA-01 "do not invent a source" discipline). Update `financialPageData()` (line 87) AND the `overview.financial` slice of `overviewPageData()` (line 165-167), which shares the same defect.
- **Why it fixes it:** The verifier evidence is "every FinancialAnalyticsPageData field is undefined at runtime → consumer defaults to zeros." A mapper that reads the RPC's actual `overview` keys makes `metrics` real; wiring the two existing chart RPCs makes `monthlyMetrics`/`breakdown.expenses`/`netOperatingIncome` real. Nothing is asserted that the data doesn't contain.
- **Risks / interactions:** Sibling within this phase: `overviewPageData()` uses the identical `jsonObject<FinancialAnalyticsPageData>` cast and must be fixed in the same pass (class-wide — see Cross-cutting). Phase 39 DATA executes before 40 and may retouch the analytics RPCs / `analytics-keys.ts`; if DATA changes `get_financial_overview`'s return shape, re-verify the mapper's key reads. - **Files touched:** src/hooks/api/use-analytics.ts, src/hooks/api/query-keys/analytics-mappers.ts

## TYPE-04 — local `Expense` type fabricates columns → table renders fake fallbacks
- **Finding:** src/hooks/api/query-keys/expense-keys.ts:277 (medium) — the local all-optional `Expense` interface declares `description` (real but never selected) plus `category`/`property_name`/`property_id` (not columns of `expenses`), so the expenses table renders "Expense" / "--" / an "other" badge on every row.
- **Root cause:** `EXPENSE_SELECT` (line 133) selects only `id, amount, expense_date, vendor_name, maintenance_request_id, status, created_at` — it omits the real `description` column (supabase.ts confirms `expenses.description text`). The local `Expense` interface (line 277-289) duplicates the canonical `ExpenseRecord = Tables<"expenses">` (core.ts:125 — a Rule-3 violation) and additionally invents `category`, `property_name`, `property_id`, none of which exist on `expenses` (the only property linkage is `expenses → maintenance_requests → units → properties`). The raw `(data ?? []) as Expense[]` casts (lines 192, 236, 261) launder the fabrication past the compiler. Consumers: expense-table.tsx renders `expense.description || "Expense"` (139, always "Expense"), `expense.property_name || "--"` (149, always "--"), `getCategoryBadge(expense.category ?? "other")` (152, always "other"); page.tsx filters/searches on `expense.category`/`expense.property_name` (26,34) which are always undefined, so the category filter matches nothing.
- **Fix (three parts):**
  1. **`description`:** add `description` to `EXPENSE_SELECT` (line 133) — real column, fixes the "Expense" fallback.
  2. **`property_name`:** enrich each query with a nested PostgREST select `maintenance_requests(units(properties(name)))` and map the joined property name into a derived `property_name` via a typed mapper `mapExpenseRow(raw)` (per CLAUDE.md's "typed mapper at every boundary").   3. **`category`:** the schema has no category column and no taxonomy; `get_expense_summary` uses `vendor_name` as the de-facto "category" grouping. Remove the fabricated `category` field. - **Why it fixes it:** Verifier evidence is "description never selected; category/property_name/property_id don't exist; casts hide it." Selecting `description`, joining `property_name`, and deleting `category` removes every fabricated field; the schema-derived view type makes the boundary honest so the fabrication can't reappear.
- **Risks / interactions:** Touches UI (expense-table.tsx, page.tsx) beyond the type file — the type fix necessarily surfaces compile errors at the fabricated reads, which is the point. The nested `properties(name)` join stays owner-scoped via the maintenance_requests join (RLS already enforces this). This overlaps Phase 41 COMP / 42 DASH territory (the expense list UI); TYPE owns the type + query correctness, and the UI edits here are the minimum to stop rendering fabricated data. Keep the `ExpenseCategoryBreakdown` summary component (fed by `get_expense_summary`, unrelated).

(B) repurpose the Category column/filter to `vendor_name` (the grouping key `get_expense_summary` already uses), keeping a filter affordance. ## TYPE-07 — custom_fields jsonb cast to DynamicField[] with no validation
- **Finding:** src/hooks/api/query-keys/template-definition-keys.ts:48 (medium) — `(data?.custom_fields ?? []) as DynamicField[]` pushes an unvalidated jsonb payload into typed land, so a malformed/legacy row crashes the dynamic-form renderer instead of failing loudly at the boundary.
- **Root cause:** `document_template_definitions.custom_fields` is `jsonb NOT NULL DEFAULT '[]'` with **no shape CHECK** (migration 20260311200000) and is owner-writable via PostgREST (RLS insert/update policies). Line 48 bare-casts it to `DynamicField[]` (`{name,label,type,...}`, dynamic-form.tsx:25-47) with no array check and no per-field validation. The consumer `useTemplateDefinition` (template-definition.ts:44 `setCustomFields(data)`) then runs `customFields.forEach(field => ... field.name ... field.type)` (line 51) and `[...baseFields, ...customFields]` (line 64). A non-array payload throws `TypeError: forEach is not a function`; entries missing `name`/`type` produce broken form fields typed as valid. This violates CLAUDE.md's "typed mapper at every boundary / Zod safeParse" rule (mapDocumentRow precedent).
- **Fix:** Add a validating boundary mapper. Create `src/lib/validation/dynamic-field.ts` exporting `dynamicFieldSchema: z.ZodType<DynamicField>` (imports the `DynamicField` type from dynamic-form.tsx — no duplicate type; the schema mirrors it: `name` non-empty string, `label` string, `type` the exact enum union, optional `placeholder/description/options/section/fullWidth/rows/itemFields/addLabel/itemLabel`) plus `parseDynamicFields(raw: unknown): DynamicField[]` that: returns `[]` if `raw` is not an array, else `safeParse`s each element and keeps only the valid ones (logging dropped entries via the frontend logger). In `template-definition-keys.ts:48`, replace the cast with `return parseDynamicFields(data?.custom_fields)`.
- **Why it fixes it:** Verifier evidence is "non-array payload crashes `customFields.forEach`; malformed entries render broken fields." `parseDynamicFields` guarantees the query resolves to a real `DynamicField[]` — non-array → `[]` (renderer shows the empty editor, no crash), malformed entries dropped (no broken fields), valid entries typed soundly. Matches the mapDocumentRow / Zod safeParse boundary pattern CLAUDE.md mandates.
- **Risks / interactions:** The save path (template-definition.ts:82 upserts `customFields`) already writes validated arrays from `CustomFieldsEditor`, so normal round-trips are unaffected; the mapper only hardens against legacy/direct-API rows. `template-definition.test.ts` asserts `customFields` equals the saved fields on load — the mapper must pass well-formed fixtures through unchanged (it does). Prefer drop-invalid over throw: this is an owner-editing surface where one bad legacy field should not blank the whole template page. Zod is already a project dependency (used across `#lib/validation/`).
- **Files touched:** src/hooks/api/query-keys/template-definition-keys.ts, src/lib/validation/dynamic-field.ts
