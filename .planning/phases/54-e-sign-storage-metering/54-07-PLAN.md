<objective>
Close the client half of D-03 ("upgrade prompt, not a raw error") for STORAGE uploads. The Plan 04 DB trigger correctly blocks over-quota uploads, but that block currently reaches nobody as an upgrade prompt: storage uploads go through the Storage API (`@supabase/storage-js`), so a rejection is a `StorageApiError` carrying only `{name,message,status,statusCode}` â€” no `.hint`/`.detail` â€” and the existing plan-limit toast (`mutation-error-handler.ts`) gates on `hint === 'plan_limit_exceeded'`, which a storage error can never satisfy. Worse, the three real upload sites discard the error entirely (document-keys.ts throws a hardcoded string; use-supabase-upload.ts/avatar capture only `error.message`). This plan (a) teaches `handleMutationError` to detect a storage-quota rejection by the trigger's stable `plan_limit_exceeded:` message prefix, (b) adds a proactive client-side pre-check (the PRIMARY UX) that shows the upgrade prompt before uploading, and wires both into all three real upload call sites.

Purpose: METER-04 / D-03 require uploads soft-enforced "with an upgrade prompt", not a raw error. The DB block (Plan 04) delivers data integrity; this plan delivers the honest UX. It depends_on 54-04 (the trigger's `plan_limit_exceeded:` message-prefix contract + the enforcement column/flag) and 54-05 (the `usageQueries.storage()` factory reused by the pre-check). It is sequenced BEFORE the Plan 06 go-flip on purpose â€” enforcement must not go live until this CTA path ships (Plan 06 depends_on 54-07).
Output: a `storage-plan-limit.ts` detector/helper, the extended `handleMutationError`, and the three wired upload sites with unit/hook tests.
</objective>
