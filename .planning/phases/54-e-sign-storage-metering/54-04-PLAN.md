<objective>
Install the real storage upload guard: a BEFORE INSERT trigger on `storage.objects` (`enforce_storage_quota`) that enforces the plan quota on the owner's pre-existing running SUM and raises the structured `plan_limit_exceeded` exception (so the client shows an Upgrade toast, not a generic error). Add the `users.storage_grandfathered_at` column and seed the `app_config` kill-flag `storage_enforcement_enabled='false'` so the trigger is created but is a NO-OP in prod until Plan 06 flips the flag. Uploads go client-direct to Storage with no edge-fn chokepoint, so a DB trigger is the only universal guard.

Purpose: METER-04 requires uploads soft-enforced against the plan quota with an upgrade prompt, never blocking reads/downloads/deletes and never a retroactive lockout. RESEARCH: `metadata->>'size'` is NULL at BEFORE INSERT, so enforce on the pre-existing SUM; a trigger (not RLS) is used because an RLS 42501 denial is genericized by the client and loses the upgrade CTA.
Output: one migration applied to prod (filename reconciled, trigger created but flag OFF), regenerated `src/types/supabase.ts`, and the METER-04 enforcement matrix appended to the storage RLS test — green.
</objective>
