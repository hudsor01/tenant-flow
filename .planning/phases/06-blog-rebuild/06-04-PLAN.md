**Purpose:** delivers BLOG-04, BLOG-05, BLOG-06 in one motion. The 12 rows are the
deliverable; the staggered publish flip is the operational follow-up.

**BLOG-05 mapping (Blocker-#2 fix):** REQUIREMENTS.md BLOG-05 reads "automated SEO + tone
check pre-publish". That requirement is satisfied by the gate machinery built in Plans 06-01
and 06-03 â€” there is no separate runtime probe in Phase 6. Specifically:
- The 9 DB validation triggers in `validate_blog_post()` (Plan 06-01) run automatically on
  every INSERT/UPDATE transitioning to `'in-review'` or `'published'`, rejecting any row that
  fails word-count, H2-count, persona phrase, slug pattern, meta length, excerpt length,
  category enum, banlist, or DocuSeal mention count.
- The n8n preflight (Plan 06-03) re-runs the same gates client-side to fail fast before HTTP.
- The integration test `tests/integration/rls/blogs-status-workflow.rls.test.ts` (Plan 06-01)
  pins each gate's rejection behavior end-to-end against prod.

The editorial flip itself (Studio UPDATE setting `status='published'`) is human-mediated â€”
that's the "tone check" review step. The "automated" requirement is the DB gate that no
content reaches `'in-review'` without passing all 9 checks.

**Blocker-#1 wiring summary:** Brief #10 (`tenantflow-vs-buildium`) passes `canonical_url:
'/compare/buildium'` in the n8n payload. The Edge Function (Plan 06-03) threads it into the
INSERT. The blogs row (Plan 06-01 column) stores it. The post page's `generateMetadata()`
(Plan 06-02) emits `alternates.canonical: post.canonical_url`, which Next.js renders as
`<link rel="canonical" href="/compare/buildium" />` inside `<head>` â€” the correct location
for canonical tags per Google's spec. The original "embed `<link>` inside markdown body"
instruction was structurally non-functional and is fully removed from this plan.
