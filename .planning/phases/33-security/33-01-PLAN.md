<objective>
Fix SEC-01 — MFA (aal2) is enforced ONLY client-side. Today the assurance level is checked as a navigation decision in `login/page.tsx` after `signInWithPassword`; `proxy.ts` gates user/is_admin/subscription but never `aal`, and layouts don't either. An aal1 session for an MFA-enrolled user reaches every private route by typing the URL, and dismissing the OTP dialog via Escape/backdrop/X keeps the aal1 session live (`onOpenChange` only closes; only the explicit Cancel button calls `signOut`).

Three locked changes (frontend/proxy, NO migration, getAll/setAll only — see 33-CONTEXT SEC-01):
1. `updateSession` (middleware.ts) also returns `assuranceLevel` from `supabase.auth.mfa.getAuthenticatorAssuranceLevel()` — this is LOCAL (decodes the JWT `aal` claim for currentLevel + derives nextLevel from the session's verified factors; zero extra round-trip since the session was just refreshed). `try/catch` → null on error (fail-secure). Return type becomes `{ user, assuranceLevel, supabaseResponse }`.
2. `proxy.ts` — after the `if (!user)` block and BEFORE the subscription allowlist, gate: if the session needs step-up (nextLevel aal2, currentLevel not aal2) → `redirectWithCookies` to `/login?redirect=<pathname>`, wrapped in `withCsp`. `/login` is public (not in `PRIVATE_ROUTE_PREFIXES`) → no redirect loop.
3. `login/page.tsx` — (a) route every dialog dismiss through sign-out via `onOpenChange`; (b) an on-mount effect that, for an aal1 enrolled session with a verified TOTP factor, opens the OTP challenge (so the proxy's `→ /login?redirect=` shows the challenge, not the password form).

The nextLevel/currentLevel decision is factored into a pure, unit-tested `requiresMfaStepUp` predicate so the proxy gate and the login on-mount effect share one source of truth. Reading only the raw JWT `aal` claim is INSUFFICIENT (can't distinguish "aal1 no-MFA = allow" from "aal1 enrolled = block") — the predicate keys on `nextLevel` from `getAuthenticatorAssuranceLevel()`.

Purpose: make MFA a server-enforced access boundary, not a client-side navigation suggestion.
Output: shared predicate + test; middleware return-type change; proxy gate; login dismiss→signOut + on-mount challenge.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated aal1 session → private route (proxy) | A password-only session for an MFA-enrolled user must be denied dashboard access server-side |
| OTP dialog dismiss → live session | Closing the challenge must not leave an authenticated aal1 session usable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-01-EoP | Elevation of Privilege | proxy.ts private-route gate | mitigate | Server-side `requiresMfaStepUp(assuranceLevel)` gate redirects any aal1-enrolled session to /login before the subscription gate; the check keys on `nextLevel` (not the raw aal claim) so non-MFA aal1 users are not falsely locked out. Enforced in the proxy, not bypassable by typing a URL. |
| T-33-01-Spoof | Spoofing | OTP dialog dismiss keeping aal1 session live | mitigate | `onOpenChange` routes Escape/backdrop/X through `handleMfaCancel` → `supabase.auth.signOut()`; a dismissed challenge no longer leaves a usable partial session. |
| T-33-01-Info | Information Disclosure | getAuthenticatorAssuranceLevel error | mitigate | Errors coerce assuranceLevel to null (fail-secure) and capture to Sentry at warning; no auth internals surface to the client. |
| T-33-01-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan. |
</threat_model>
