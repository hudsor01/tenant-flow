## SEC-01 — MFA (aal2) not enforced server-side
**Confirmed:** assurance level is checked ONLY client-side in `login/page.tsx` (a navigation decision post-`signInWithPassword`). `proxy.ts` gates user/is_admin/subscription but never aal; layouts don't either. An aal1 session for an MFA-enrolled user reaches every private route by typing the URL. Dismissing the OTP dialog via Escape/backdrop/X keeps the aal1 session live (`onOpenChange` only closes; only the explicit Cancel button calls `signOut`).
**Decisions (frontend/proxy, no migration; getAll/setAll only):**
1. `src/lib/supabase/middleware.ts` — `updateSession` also returns `assuranceLevel` from `await supabase.auth.mfa.getAuthenticatorAssuranceLevel()` (this is LOCAL — decodes the JWT `aal` for currentLevel + derives nextLevel from `session.user.factors`; zero extra round-trip since the session was just refreshed). `try/catch` → null on error (fail-secure). Return type becomes `{ user, assuranceLevel, supabaseResponse }`.
2. `src/proxy.ts` — after the `if (!user)` block and BEFORE the subscription allowlist, gate: if `assuranceLevel?.nextLevel === "aal2" && assuranceLevel.currentLevel !== "aal2"` → `redirectWithCookies` to `/login?redirect=<pathname>` (wrapped in `withCsp`). `/login` is public → no redirect loop. Reading only the raw JWT `aal` claim is INSUFFICIENT (can't distinguish "aal1 no-MFA=allow" from "aal1 enrolled=block") — must use `nextLevel` from `getAuthenticatorAssuranceLevel()`.
3. `src/app/(auth)/login/page.tsx` — (a) route every dialog dismiss through sign-out: `onOpenChange={(next) => { if (!next) void handleMfaCancel(); }}` (Radix does NOT fire onOpenChange on controlled `setShowMfaDialog(false)`, so `handleMfaSuccess` won't double-signOut); (b) on-mount effect: if `getUser()` && `nextLevel==='aal2' && currentLevel!=='aal2'` && a verified TOTP factor exists → set the factorId + pending redirect (validate via `isValidRedirect`) + open the OTP dialog (so the proxy's `→/login?redirect=` shows the challenge, not the password form).
- No change to `mfa-verification-dialog.tsx` needed (the login-page onOpenChange covers all dismiss paths).

## SEC-02 — storage images CSP-blocked on private routes
**Confirmed:** both CSP defs ship `img-src 'self' data: blob:` (no storage origin), NO `media-src`, NO `frame-src`. Private-route media loads via raw cross-origin `<img>`/`<video>`/`<iframe>` at the storage host `https://bshjmbshupiibfiewpxb.supabase.co` (signed/public URLs). Blocked: maintenance photos (`<img>`) + videos (`<video>` → media-src), inspection photos, document image preview (`<img>`) + PDF preview (`<iframe>` → frame-src), avatar (`<img>`). `connect-src` ALREADY uses `*.supabase.co` — mirror that wildcard style. Property images are fine (next/image, same-origin).
**Decision:** in BOTH `src/proxy.ts` (`buildNonceCsp`) and `vercel.json`, extend: `img-src 'self' data: blob: https://*.supabase.co`; add `media-src 'self' data: blob: https://*.supabase.co`; add `frame-src 'self' https://*.supabase.co` (for the PDF iframe). Do NOT introduce a bare `https:`. Keep the two files' CSP strings in sync.

## SEC-03 — public shared-cache header on auth-walled /properties + dead rules
**Confirmed:** `vercel.json` (~lines 128-158): `/properties/(.*)` sets `Cache-Control: public, s-maxage=300, stale-while-revalidate=3600` + `Vary: Authorization, Cookie` (Vercel CDN does NOT honor Vary:Cookie for key separation → owner A's property-detail HTML can be served to owner B within the window). `/properties` is auth-walled (in `PRIVATE_ROUTE_PREFIXES`). `/manage` + `/tenant` are dead exact-match rules (no such routes; `/tenants` plural doesn't match).
**Decision:** replace the `/properties/(.*)` block with a single `Cache-Control: private, no-store, must-revalidate` header (defense-in-depth, auditable) and DELETE the `/manage` + `/tenant` blocks. Preserve all legit caching (static assets, marketing, OG images, blog) — touch only these three blocks.

## SEC-04 — leases UPDATE re-points to another owner's unit/tenant (MIGRATION)
**Confirmed against live prod:** `leases_update_owner` WITH CHECK pins only `owner_user_id` — NEW `unit_id`/`primary_tenant_id` unvalidated, so owner A can `UPDATE ... SET unit_id=<B's unit>, primary_tenant_id=<B's tenant>`. `leases_insert_owner` validates `unit_id` but NOT `primary_tenant_id` (sibling gap). `lease_tenants` insert/update validate only the parent lease, not `tenant_id` (parallel gap). `units.owner_user_id` (NOT NULL) + `tenants.owner_user_id` are owner-scoped; `get_current_owner_user_id() = auth.uid()`.
**Decision:** RLS WITH CHECK (cleaner than a trigger; mirrors the proven INSERT pattern). One migration recreating four policies (CREATE OR REPLACE via drop+create):
- `leases_insert_owner`: add `primary_tenant_id IN (owner's tenants)` alongside the existing owner + unit checks.
- `leases_update_owner`: WITH CHECK = owner + `unit_id IN (owner's units)` + `primary_tenant_id IN (owner's tenants)`; USING stays `owner_user_id = owner`.
- `lease_tenants_insert_owner` / `lease_tenants_update_owner`: add `EXISTS (tenants t WHERE t.id = tenant_id AND t.owner_user_id = auth.uid())` alongside the parent-lease check.
(Exact DDL captured in the understand report — the executor applies it verbatim.) **MIGRATION = yes.** RLS-integration test in `tests/integration/rls/`: ownerA lease/lease_tenants cannot be re-pointed to ownerB's unit/tenant (dual-client, expect RLS rejection). NOTE: if any pre-existing prod row is already cross-owner, a plain WITH CHECK would block future updates to it — the understand report says none exist; verify via a count query before applying, and if any exist use the `IS DISTINCT FROM old` trigger variant instead.

## SEC-05 — e-signature audit trail not tamper-proof (MIGRATION)
**Confirmed against live prod:** the only BEFORE-UPDATE guard on `leases` is `reject_signed_lease_term_edits` (locks 6 FINANCIAL columns only; its comment says signature columns are intentionally unlocked). RLS is row-level not column-level, so an owner can PATCH `tenant_signature_name='Someone Else'` (or a doctored IP/UA/method/consent/hash) after the tenant signs; `finalizeSignedLease`/`buildLeasePdfData` regenerate the signed PDF live from these columns (`buildLeasePdfData:234 name: record.tenantSignatureName ?? record.tenantName`), so the forgery renders onto the signature block + audit certificate. Distinct from the term-lock.
**State machine per audit column:** `null→value` ALLOWED (sign), `value→null` ALLOWED (cancel/revert), `value→different-non-null` REJECTED (tamper), `value→same` allowed (no-op).
**Decision:** BEFORE UPDATE trigger `reject_signature_audit_tampering()` (SECURITY DEFINER, `SET search_path = public`) — for EACH of the 12 audit columns, reject when `old.<c> IS NOT NULL AND new.<c> IS NOT NULL AND new.<c> IS DISTINCT FROM old.<c>`, RAISE (mirror the term-lock trigger's RAISE convention/errcode). Columns: `owner_signed_at, owner_signature_ip, owner_signature_user_agent, owner_signature_method, owner_signature_consent_at, tenant_signed_at, tenant_signature_ip, tenant_signature_user_agent, tenant_signature_method, tenant_signature_name, tenant_signature_consent_at, signed_document_hash`. (No `owner_signature_name` column — owner name renders from `users.full_name`, outside the threat.) `CREATE TRIGGER ... BEFORE UPDATE ON public.leases FOR EACH ROW EXECUTE FUNCTION reject_signature_audit_tampering()`. **MIGRATION = yes.** MUST NOT block the legit sign (`sign_lease_with_token` null→value) or cancel (`lease-signature` cancel value→null) flows — cross-check those edge functions + `useSignLeaseAsOwnerMutation`/`useCancelSignatureRequestMutation`. Integration test: after tenant-sign, a PATCH changing `tenant_signature_name` to a different value is rejected; null→value (sign) and value→null (cancel) succeed.
