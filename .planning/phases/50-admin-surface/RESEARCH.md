## ADMIN-01 — Admin analytics RPC errors silently swallowed
- **Finding:** src/app/(admin)/admin/analytics/page.tsx:34 (medium) — the three admin analytics RPC results are read only via `.data`; `.error` is never checked, so any RPC failure renders as a genuine-empty state with no Sentry capture.
- **Root cause:** Two independent defects converge here.
  1. **No observability.** `Promise.all` returns `{ data, error }` for each RPC; the page destructures `[deliverabilityResult, funnelResult, gateResult]` (line 34) and only ever inspects `.data` (lines 47-65). A failing RPC returns `{ data: null, error: {...} }`; `Array.isArray(null)` and the non-object null branch both fall through to `[]`/`null`, which the child components render as their own "No … data yet" `Empty` states (deliverability-table.tsx:44, gate-conversion-table.tsx:49, funnel-chart.tsx:52). An outage is indistinguishable from empty data for both the admin and Sentry. The sibling admin blog page (blog/page.tsx:26-28) already `Sentry.captureException`s its fetch error — this page does not.
  2. **A real, self-inflicted funnel failure mode.** `funnelTo` is computed as JS `now().toISOString()` on the Vercel server (line 32) and passed as `p_to`. `get_funnel_stats` raises `'Invalid window: p_to (%) cannot be in the future'` when `p_to > now()` measured against the *Postgres* clock (20260419040000_drop_first_rent_funnel_step.sql:38, `p_to timestamptz default now()` in 20260415193248). Any Vercel/DB clock skew where Vercel is ahead throws — a swallowed 500 on every request until clocks resync.
- **Fix:** Two parts, both root-cause:
  1. **Eliminate the clock-skew failure at the source.** Stop passing a client-computed `p_to`. The RPC already defaults `p_to timestamptz default now()` (DB clock). Call `supabase.rpc("get_funnel_stats", { p_from: funnelFrom })` with `p_from` only — PostgREST omits the unspecified arg and Postgres applies the `now()` default, so `p_to > now()` can never be true. Delete the `funnelTo`/`now.toISOString()` `p_to` computation (lines 28,32,40).
  2. **Add observability + visible degradation.** Import `* as Sentry from "@sentry/nextjs"`. After the `Promise.all`, for each of the three results, if `.error` is set call `Sentry.captureException(result.error, { tags: { page: "admin-analytics", rpc: "<get_deliverability_stats|get_funnel_stats|get_gate_conversion_stats>" } })` (mirrors the blog page's tag pattern). Then surface the error to the admin so it is not masked as empty: derive a per-section `errored` boolean and, when set, render a compact inline notice in place of the table/chart — e.g. `<p className="text-sm text-destructive-text">Failed to load — refresh to retry.</p>` (vivid-token rule: `text-destructive-text` for text, not `text-destructive`). Keep the existing `[]`/`null` mapping as the non-errored fallback.
- **Why it fixes it:** Dropping the client `p_to` removes the exact `p_to > now()` RAISE the verifier reproduced, so the funnel RPC no longer 500s on clock skew. Checking `.error` + Sentry per result matches the sibling blog-page pattern the finding cites and makes every future RPC outage observable; the inline `text-destructive-text` notice makes an outage visually distinct from genuine-empty for the admin, closing the "masked as No data yet" gap.
- **Risks / interactions:** Adding `Sentry` import + `.error` branches is additive; no behavior change on the success path. The inline error notice must use `text-destructive-text` (not `text-destructive`) per the vivid-token WCAG note. Confirm `get_funnel_stats` still returns the intended ~90-day cohort: `p_from` is still passed explicitly (90-day window preserved); only `p_to` defaults to DB-now, which is the correct upper bound. No other caller of these three RPCs — change is local to this page. No cross-phase file overlap (Phase 48 SEO-14 touches deliverability-table.tsx, not this page).
- **Files touched:** src/app/(admin)/admin/analytics/page.tsx

## ADMIN-02 — `blogs.word_count` never populated → every draft shows "0 words"
- **Finding:** src/app/(admin)/admin/blog/blog-review-client.tsx:75 (low) — review queue renders `{draft.word_count ?? 0} words`, but no writer populates `blogs.word_count`, so every 1,200–3,000-word draft shows "0 words" and the generated `reading_time` column is NULL too.
- **Root cause:** `blogs.word_count` is a nullable `INTEGER` (20251209120000_create_blogs_table.sql:13) that no code path ever writes. The n8n-blog-ingest insert row omits it (index.ts:374-383). The `validate_blog_post` BEFORE trigger (20260510214935:33) computes a local `v_word_count := array_length(regexp_split_to_array(coalesce(NEW.content,''), '\s+'),1)` purely to gate the 1200..3000 range check (line 34) and never assigns it to `NEW.word_count`. So `word_count` is NULL on every row, `blog-review-client.tsx:75` coalesces to 0, and `reading_time GENERATED ALWAYS AS (GREATEST(1, word_count / 200))` (create migration:14) is NULL sitewide (it is also selected in BLOG_REVIEW_COLUMNS and BlogListItem, so public blog cards lose reading-time too).
- **Fix (DB migration, root-cause — Direction A):** New migration `supabase/migrations/<ts>_populate_blog_word_count.sql` that `CREATE OR REPLACE FUNCTION public.validate_blog_post()` identical to 20260510214935 but adds `NEW.word_count := v_word_count;` immediately after the range check passes (inside the existing `IF (INSERT|UPDATE-into-review/published)` block, after line 35's RAISE guard). Because `reading_time` is a `GENERATED ALWAYS` column, Postgres recomputes it from the final `NEW.word_count` after the BEFORE trigger — so both columns populate for every future in-review/published write (ingest INSERT path included, since its block condition `TG_OP='INSERT' AND NEW.status IN ('in-review','published')` is already true). Include a one-time backfill in the same migration: `UPDATE public.blogs SET word_count = array_length(regexp_split_to_array(coalesce(content,''), '\s+'),1) WHERE word_count IS NULL;` — this direct column write recomputes `reading_time` and does NOT re-fire the validation block (the block's UPDATE arm requires `OLD.status NOT IN ('in-review','published')`, which is false for already-published rows). Preserve the REVOKE + trigger recreate from the original migration. After applying via MCP, reconcile the filename timestamp per the migration-mcp-prod-drift rule. No edge-function change needed (the trigger covers the ingest path), so the CLI-401 owner-run deploy is not required for this fix. `bun run db:types` after apply (word_count/reading_time types are unchanged — both already `number | null` — so likely a no-op, but run to confirm).
- **Why it fixes it:** The verifier confirmed the only gating computation (`v_word_count`) is discarded; assigning it to `NEW.word_count` makes the value the review UI already reads truthful, and the generated `reading_time` derives for free. The backfill repairs the existing NULL rows so historical published posts also gain reading-time. `blog-review-client.tsx` needs no change — `draft.word_count ?? 0` renders the real count once the column is non-NULL.
- **Risks / interactions:** Generated-column recompute on a BEFORE trigger is standard and safe (base column assigned before the row is finalized). The backfill UPDATE touches every NULL-word_count blog row once; bound is the blog table size (paused factory left ~218 rows) — trivial. Confirm the regex whitespace count matches the gate's own count (it does — same expression) so a backfilled row can never violate the 1200..3000 gate it already passed. No app-code overlap with other phases.
- **Files touched:** supabase/migrations/&lt;new-ts&gt;_populate_blog_word_count.sql
- **Decision:** Direction A (populate `NEW.word_count` in the trigger + backfill) chosen over Direction B (drop `word_count`/`reading_time` from BLOG_REVIEW_COLUMNS + BlogReviewItem + mapper + the "{n} words" render). A is root-cause: it also repairs `reading_time` NULL sitewide (public blog cards), whereas B only hides the admin symptom and leaves reading-time broken. B is lower blast radius but discards a metric the schema already models and the public list already selects.

## ADMIN-03 — Reject archives a draft on one click, no confirmation
- **Finding:** src/app/(admin)/admin/blog/blog-review-client.tsx:104 (low) — `onClick={() => reject.mutate({ id: draft.id })}` archives the draft immediately; no AlertDialog, no undo, no archived-post restore surface.
- **Root cause:** The Reject button (lines 101-108) sits directly beside Approve and fires the destructive mutation with no confirmation. `rejectBlogPost` (blog-publish.ts:147-150) flips `status` to `'archived'`; the admin area has only `analytics` + `blog` (review queue) and no view that lists or restores archived rows, so a misclick silently removes a queued draft with no recovery path.
- **Fix:** Wrap Reject in an AlertDialog confirmation following the codebase convention (property-units-delete-dialog.tsx). `blog-review-client.tsx` is already a client component. In `BlogReviewRow`, add `const [confirmReject, setConfirmReject] = useState(false)`; change the Reject button's `onClick` to `() => setConfirmReject(true)`; render an `<AlertDialog open={confirmReject} onOpenChange={setConfirmReject}>` with title "Reject this draft?", a description noting it archives the post and is not undoable from this screen, `<AlertDialogCancel>`, and an `<AlertDialogAction className="bg-destructive text-destructive-foreground hover:bg-destructive/90" onClick={() => reject.mutate({ id: draft.id })}>` (the destructive-styled confirm from the convention file). Import the AlertDialog primitives from `#components/ui/alert-dialog`.
- **Why it fixes it:** Interposes the exact AlertDialog gate the finding cites as the codebase convention, so the single-click destroy path the verifier confirmed (line 104 → immediate archive) now requires explicit confirmation. Approve stays one-click (non-destructive, publishable/reversible).
- **Risks / interactions:** Purely additive client state; `reject.mutate` still runs through the same mutation + Sentry error path. Keep `disabled={isPending}` on the trigger button so a confirm-then-spam is blocked. No overlap with ADMIN-02 (a DB migration) or ADMIN-04 (query bound). Leave Approve un-gated to preserve the fast approve flow.
- **Files touched:** src/app/(admin)/admin/blog/blog-review-client.tsx

## ADMIN-04 — Unbounded review-queue list query (two sites)
- **Finding:** src/app/(admin)/admin/blog/page.tsx:24 (low) — the `blogs` review-queue select filters `status='in-review'` and orders but has no `.limit()`/`.range()`, violating the CLAUDE.md "all list queries MUST have `.limit()` or `.range()`" rule; each row carries full 1,200–3,000-word `content`.
- **Root cause:** The bound is missing in **both** copies of the query: the server-side fetch (page.tsx:20-24) and the `blogQueries.reviewQueue()` factory (blog-keys.ts:173-177). They are byte-identical (same `BLOG_REVIEW_COLUMNS`, same `.eq("status","in-review").order("created_at",{ascending:false})`) because the factory seeds the client `useQuery` via `initialData` from the server fetch (blog-review-client.tsx:30-33) — so any bound must be applied identically to both or the SSR seed diverges from the client refetch.
- **Fix (class-wide, both sites):** Add a shared `const REVIEW_QUEUE_LIMIT = 50` to blog-keys.ts, export it, and append `.limit(REVIEW_QUEUE_LIMIT)` to **both** the `reviewQueue()` factory query (after `.order(...)`) and the page.tsx server fetch (import the const). 50 comfortably exceeds any realistic in-review backlog (the paused factory's ~218 rows are `published`/`archived`, not `in-review`) while bounding the full-content payload. Keep both queries otherwise identical so `initialData` stays consistent.
- **Why it fixes it:** Adds the `.limit()` the rule requires to the exact unbounded query the verifier flagged at both cited locations (page.tsx:20-24 and blog-keys.ts:173-177), bounding row count and full-content transfer.
- **Risks / interactions:** The limit must match in both files or the SSR-seeded `initialData` (first 50) mismatches the client refetch, causing a hydration/refetch flash — a single shared const prevents drift. `content` stays in the row because the inline preview (blog-review-client.tsx:124) renders `draft.content`; dropping content would force a second fetch and is out of scope. If an in-review backlog could ever exceed 50, a follow-up would add `.range()` pagination — noted, not needed now. No cross-phase overlap.
- **Files touched:** src/app/(admin)/admin/blog/page.tsx, src/hooks/api/query-keys/blog-keys.ts

## ADMIN-05 — Bare `/admin` 404s; no inter-page nav
- **Finding:** src/app/(admin)/layout.tsx:23 (low) — the (admin) group has only `/admin/analytics` and `/admin/blog`; there is no `(admin)/admin/page.tsx` and no `/admin` redirect, so a bare `/admin` from an authenticated admin 404s. The header offers no links between the two admin pages.
- **Root cause:** No index route maps to `/admin` (the `admin/` folder has only `analytics/` and `blog/` subfolders and no `page.tsx`), and `next.config.ts`/`vercel.json` have no `/admin` redirect. `proxy.ts:336` only bounces non-admins, so a bare `/admin` request from an admin passes proxy + layout gates and hits the 404. The header (layout.tsx:38-42) renders only an `<h1>Admin</h1>` with zero nav links, so `/admin/blog` is undiscoverable.
- **Fix (two parts):**
  1. **Index route.** Create `src/app/(admin)/admin/page.tsx` — a Server Component whose body is `redirect("/admin/analytics")` (import `redirect` from `next/navigation`; optionally add `export const metadata: Metadata = { robots: { index: false, follow: false } }` for consistency, though the layout already sets robots). It renders inside the (admin) layout, so the `is_admin` auth-wall still runs before the redirect. Analytics is the de-facto index (layout.tsx:23 already redirects login there).
  2. **Header nav.** In layout.tsx replace the lone `<h1>` block with the `<h1>Admin</h1>` plus a `<nav aria-label="Admin">` containing `<Link href="/admin/analytics">Analytics</Link>` and `<Link href="/admin/blog">Blog Review</Link>` (import `Link` from `next/link`; Links are server-compatible so the layout stays a Server Component). Style with existing tokens (`text-sm text-muted-foreground hover:text-foreground`); no active-state highlight (would force a client component via `usePathname` — avoid).
- **Why it fixes it:** The redirecting `page.tsx` gives `/admin` a real route that lands admins on analytics instead of 404 — exactly the "redirecting page.tsx at (admin)/admin/" the finding prescribes. The header links make `/admin/blog` reachable, closing the "zero nav links" gap the verifier confirmed.
- **Risks / interactions:** `redirect()` in a route-group index page is standard Next.js 16 and does not conflict with the `(admin)/loading.tsx` fallback. Keeping the layout server-side means no `usePathname`/client boundary. Non-admins hitting `/admin` still get bounced to `/dashboard` by the layout before the redirect matters. Shares layout.tsx with ADMIN-06 — sequence the two layout edits into one change.
- **Files touched:** src/app/(admin)/admin/page.tsx (new), src/app/(admin)/layout.tsx
- **Decision:** A redirecting `page.tsx` inside the (admin) group is chosen over a `next.config.ts` `/admin` redirect. Both resolve the 404; the page.tsx keeps the request inside the admin auth-wall (defense-in-depth layer 2) and co-locates the index with the route group, matching the finding's own recommendation. Alternative (config redirect) would fire before routing and bypass the layout gate — acceptable but less consistent with the layered-auth design.

## ADMIN-06 — Admin `<main>` missing `id="main-content"`, breaks skip link
- **Finding:** src/app/(admin)/layout.tsx:43 (low) — the admin shell renders bare `<main>{children}</main>`; the root layout's mandated skip-to-content link targets `#main-content`, which exists only in page-layout.tsx and app-shell.tsx, neither used on `/admin` — so on every `/admin/*` page the skip link is a dead anchor.
- **Root cause:** `src/app/layout.tsx:99-104` renders one global skip link (`href="#main-content"`) for all routes; the id must be set on each route's `<main>`. The (admin) layout omits it, so keyboard/screen-reader users activating the skip link on `/admin/analytics` and `/admin/blog` jump nowhere.
- **Fix:** Change line 43 to `<main id="main-content">{children}</main>`. One-line, matching app-shell.tsx:303 and page-layout.tsx:59.
- **Why it fixes it:** Restores the anchor the global skip link targets on the two admin routes, exactly the missing id the verifier confirmed. No visual change.
- **Risks / interactions:** None — additive id attribute. Shares the file with ADMIN-05 (header nav) — bundle both layout.tsx edits together. `id="main-content"` is unique per rendered page (only one layout wraps a given route), so no duplicate-id concern.
- **Files touched:** src/app/(admin)/layout.tsx

## ADMIN-07 — GateConversionTable carries an unnecessary `"use client"`
- **Finding:** src/components/admin/gate-conversion-table.tsx:1 (low) — the component declares `"use client"` but uses no hooks, event handlers, or browser APIs; it is a pure presentational table over serializable props from the server-rendered analytics page.
- **Root cause:** The directive is a false requirement. The component imports only `Empty`/`Table` primitives (both server-compatible — verified: table.tsx and empty.tsx carry no `"use client"`), a `lucide-react` icon, `cn`, `Intl.NumberFormat`, and an array `.sort()` over props. Its sole caller is the async Server Component analytics page (page.tsx:85). The directive needlessly creates a client boundary, forcing prop serialization and shipping/hydrating the component + table.tsx/empty.tsx into the client bundle, violating "Server Components by default."
- **Fix:** Delete line 1 (`"use client";`) from gate-conversion-table.tsx. The component becomes a Server Component with zero code changes — all dependencies are server-safe.
- **Why it fixes it:** Removes the boundary the verifier confirmed unnecessary; the component renders identically server-side, dropping it (and its Table/Empty imports) from the client bundle.
- **Risks / interactions:** Minimal — no runtime-client feature is used. `text-destructive-text` class usage is CSS-only, unaffected. **Sibling / phase interaction:** `deliverability-table.tsx` is the byte-identical case and is tracked as **SEO-14 (Phase 48)**, which executes before Phase 50 — its `"use client"` is removed there, so do NOT re-fix it here (verify it is already gone when Phase 50 lands). `funnel-chart.tsx` legitimately needs `"use client"` (it uses `next/dynamic` with `ssr:false`) — leave it. `funnel-renderer.tsx` (recharts) also stays client.
- **Files touched:** src/components/admin/gate-conversion-table.tsx
