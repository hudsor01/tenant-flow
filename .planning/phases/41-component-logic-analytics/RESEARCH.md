## COMP-01 — Financial analytics stat cards divide dollars by 100 (100x understatement)
- **Finding:** `src/app/(owner)/analytics/financial/_components/financial-overview-stats.tsx:35` (high) — `<NumberTicker value={metrics.totalRevenue / 100} />` (and `netIncome`/`cashFlow` at lines 60/114) understates dollar metrics 100x.

Values are whole dollars (integer-derived), so `NumberTicker` with no `decimalPlaces` is correct; no reformatting needed. (The reference `financials-summary-stats.tsx` switched to a 2-decimal `formatUsd`, but the source there can carry cents; here the source is whole dollars so keeping `NumberTicker` is fine and matches the finding's stated fix "pass the dollar values through unchanged.")
- **Why it fixes it:** Verifier confirmed the RPC returns dollars from integer-dollar columns; removing the `/ 100` makes the displayed value equal the true dollar amount, eliminating the 100x understatement. This is one member of the v8.0 100x money class; see cross-cutting.
- **Risks / interactions:** Latent today because `financialPageData()` casts the RPC's `{overview,highlights}` shape onto `FinancialAnalyticsPageData` (which expects `{metrics,breakdown,...}`), so `metrics` is `undefined` and `page.tsx` falls back to all-zero defaults → the card shows `$0`. No edit conflict: COMP-01 touches only the presentation component. No behavior change while `metrics` stays zero.

## COMP-03 — Quarterly income statement builds invalid `-31` dates (breaks Q2/Q3)
- **Finding:** `src/app/(owner)/financials/income-statement/page.tsx:29` (high) — `end_date: `${yearNum}-${quarterEnd...}-31`` produces `2026-06-31` (Q2) and `2026-09-31` (Q3), which Postgres rejects (SQLSTATE 22008) when passed as `get_expense_summary`'s `p_end_date date`, rendering `IncomeStatementPageError`.
- **Root cause:** Hardcoded `-31` day for the quarter-end string instead of computing the real last day of `quarterEnd` month. The monthly branch (line 33) already does it correctly with `new Date(yearNum, currentMonth, 0).getDate()`; the quarterly branch was never updated.
- **Fix:** Replace line 29's `-31` literal with the computed last day, mirroring the monthly branch and the cash-flow sibling (`cash-flow/page.tsx:32`):
  `end_date: `${yearNum}-${quarterEnd.toString().padStart(2, "0")}-${new Date(yearNum, quarterEnd, 0).getDate().toString().padStart(2, "0")}``
  (`new Date(yearNum, quarterEnd, 0)` where `quarterEnd` is the 1-based end month yields the last day of that month: e.g. `new Date(2026, 6, 0)` = June 30, `new Date(2026, 9, 0)` = Sept 30, `new Date(2026, 12, 0)` = Dec 31.)
- **Why it fixes it:** Verifier confirmed the string flows into `get_expense_summary` `p_end_date date` and the invalid date raises 22008. Emitting a valid month-end (30/31/28/29) makes the cast succeed for every quarter, so Quarterly renders instead of erroring during Q2/Q3.
- **Risks / interactions:** Pure date-string arithmetic; no schema/query-shape change. Quarter ends are Mar/Jun/Sep/Dec, so leap-year Feb never applies. Only Q2/Q3 broke (Q1 end month 03 and Q4 end month 12 both accept `-31`); the computed form is correct for all four. The sibling cash-flow page is already correct — no shared helper to refactor, but the two could later be unified into one `quarterRange(year)` util (not required now).
- **Files touched:** src/app/(owner)/financials/income-statement/page.tsx

## COMP-13 — Render-time in-place `.sort()` mutates a TanStack Query cache array
- **Finding:** `src/components/reports/sections/year-end-report-section.tsx:210` (low) — `yearEndData?.expenseByCategory.sort((a,b) => b.amount - a.amount)` sorts in place during render; `yearEndData` is the raw `useYearEndSummary` cache entry (no `select`/clone), so it mutates the shared cache array.
- **Root cause:** `Array.prototype.sort` mutates in place; called directly on cache-owned data during render — a Rules-of-React violation, doubly unsafe under `reactCompiler: true`.
- **Fix:** Copy before sorting, matching the existing financials-page pattern (`[...monthlyMetrics].sort(...)`). Change lines 209-211 to:
  ```
  {yearEndData?.expenseByCategory &&
    [...yearEndData.expenseByCategory]
      .sort((a, b) => b.amount - a.amount)
      .map((cat, i) => ( … ))}
  ```
  (Guard the spread behind the optional chain so `yearEndData === undefined` still short-circuits to nothing.)
- **Why it fixes it:** Verifier confirmed the array is the un-cloned cache entry. Spreading produces a fresh array; the in-place `sort` then mutates the copy, leaving the cache immutable — structural sharing/change detection and any other consumer see stable data.
- **Risks / interactions:** Pure copy-then-sort; identical render output. No dependency on other phases.
- **Files touched:** src/components/reports/sections/year-end-report-section.tsx
- **Decision:** `[...arr].sort(...)` vs `arr.toSorted(...)` are equivalent non-mutating options. **Recommend `[...arr].sort()`** — it matches the established `[...monthlyMetrics].sort(...)` pattern already in the financials page and carries no dependency on the ES2023 lib target that `toSorted` requires.

### Bug class B — date-only column parsed as UTC midnight (COMP-05, COMP-08, COMP-09, COMP-12)
Four sites parse a Postgres `date` string with `new Date(ymd)` (UTC midnight) or format it without `timeZone: "UTC"`, shifting the calendar day back one for negative-UTC-offset (US) users. **Canonical fixes already exist in `#lib/formatters/date`:** `parseLocalYmd` (local-midnight parse, for comparisons — COMP-08, COMP-12) and `formatDate` (renders with `timeZone: "UTC"` — COMP-05, COMP-09). No new helper needed. Recommend a phase-wide confirmation sweep `rg -n "new Date\([^)]*(?:expense_date|end_date|start_date|scheduled_date|_date)" src` to catch any un-enumerated date-only siblings before closing the phase; fix only clear date-only-column siblings.
