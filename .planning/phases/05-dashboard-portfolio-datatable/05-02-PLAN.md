<objective>
Ship DT-01 (the assembled replacement shell) + DT-04 (faceted status filter) + DT-05 (column visibility) + DT-06 (always-on virtualization) + DT-07 (grid/table toggle) as a self-contained `PortfolioDataTable` component — built but NOT yet mounted. It consumes the Wave-1 outputs: `useClientDataTable` (05-01a) for state (now with controlled `columnVisibility`) and `portfolioColumns` + `getAriaSort` (05-01b) for the column/aria-sort/filter/label metadata. It composes the already-vendored `DataTableFacetedFilter`, `DataTableViewOptions`, and `DataTablePagination`, virtualizes the tbody via `useVirtualizer` over the post-pagination page rows, and FORCES the grid view on mobile (<=375px) per the UI-SPEC.

Purpose: this is the visual heart of the swap. The risks/findings it defuses:
1. **aria-sort on the correct element (B-1)** — `aria-sort` is only valid on `role="columnheader"` = the `<th>`/`<TableHead>`. The vendored `DataTableColumnHeader` renders a `<button>` flexRendered INSIDE the `<th>`, so it can never BE the `<th>`. This component renders `<TableHead aria-sort={getAriaSort(column)}>` (the 05-01b helper) so the attribute lands on the actual `<th>`; keyboard sort (Tab + Enter/Space) stays on the header button.
2. **Controlled column visibility (B-2)** — `columnVisibility` + `onColumnVisibilityChange` are passed straight into `useClientDataTable`, which Plan 05-01a already accepts as controlled props. There is NO off-plan edit to the frozen 05-01a hook.
3. **Virtualization + sticky header + pagination** — combine an absolute-positioned virtualized `<tbody>` (`getTotalSize()` height) with a sticky `<thead>` and the pagination footer (leases-table pattern), measured to the UI-SPEC min (h-12 / 48px). Always-on (D-2): one code path, no threshold branch.
4. **Grid + mobile read the same nuqs state** (D-5 + W-4) — when `viewMode === "grid"` OR the viewport is `<=375px` (forced), render the card grid off the SAME `table.getRowModel()` rows (already filtered/sorted/paged by the table), not a parallel array.
5. **Search parity (W-3)** — search filters the Property column via the pinned nuqs key `"property"`, whose name||address `filterFn` (Plan 05-01b) matches name OR address.
6. **State ownership** — the component is a pure consumer: it receives `data`, `viewMode`/`setViewMode`, and `columnVisibility`/`onColumnVisibilityChange` as props. It does NOT import `dashboard-store` or `dashboard-presets-store` (Plan 05-03 owns wiring those).

NO mount in `dashboard.tsx` (Plan 05-03). NO presets store, NO nuqs page-wiring beyond what `useClientDataTable` already provides internally. NO deletion of the hand-rolled files (Plan 05-03).
