Rewrite `apps/frontend/src/hooks/api/query-keys/inspection-keys.ts` replacing all `apiRequest` calls with `supabase.from('inspections')` PostgREST direct calls.

Follow the pattern established in `property-keys.ts` (read that file first).

**INSPECTION_SELECT_COLUMNS constant (for list queries — no rooms):**
`'id, lease_id, property_id, unit_id, owner_user_id, inspection_type, status, scheduled_date, completed_at, tenant_reviewed_at, overall_condition, owner_notes, created_at, updated_at'`

**INSPECTION_DETAIL_SELECT (for detail — includes rooms and photos):**
`'id, lease_id, property_id, unit_id, owner_user_id, inspection_type, status, scheduled_date, completed_at, tenant_reviewed_at, tenant_signature_data, overall_condition, owner_notes, tenant_notes, created_at, updated_at, inspection_rooms(id, room_name, room_type, condition_rating, notes, created_at, updated_at, inspection_photos(id, inspection_room_id, inspection_id, storage_path, file_name, file_size, mime_type, caption, uploaded_by, created_at))'`

**`inspectionQueries.list()`:**
- `supabase.from('inspections').select(INSPECTION_SELECT_COLUMNS + ', properties(name, address_line1), units(name)', { count: 'exact' }).eq('owner_user_id', userId).order('created_at', { ascending: false })`
- The `InspectionListItem` type in shared types expects `property_name`, `unit_name`, `room_count` (denormalized fields from the NestJS query). With PostgREST, join to `properties` and `units` and remap: `property_name: row.properties?.name ?? ''`, `unit_name: row.units?.name ?? null`
- For `room_count`: a separate count query per inspection would be N+1. Instead, use a PostgREST aggregate: `.select(INSPECTION_SELECT_COLUMNS + ', properties(name, address_line1), units(name), inspection_rooms(count)', ...)` and map `room_count: row.inspection_rooms?.[0]?.count ?? 0` (PostgREST count syntax)
- Actually, the cleaner approach: use `.select(..., 'inspection_rooms(id)', ...)` and map `room_count: row.inspection_rooms?.length ?? 0` — simpler and works for lists with reasonable room counts
- RLS will filter by `owner_user_id = auth.uid()` automatically — no need for manual userId filter

**`inspectionQueries.byLeaseQuery(leaseId)`:**
- `supabase.from('inspections').select(INSPECTION_SELECT_COLUMNS).eq('lease_id', leaseId).order('created_at', { ascending: true })`
- Enabled only when leaseId is truthy

**`inspectionQueries.detailQuery(id)`:**
- `supabase.from('inspections').select(INSPECTION_DETAIL_SELECT).eq('id', id).single()`
- Returns `Inspection` type from shared types (with rooms and photos nested)
- The nested inspection_photos need `publicUrl` appended: after fetching, transform each photo's `storage_path` to a public URL using `supabase.storage.from('inspection-photos').getPublicUrl(photo.storage_path).data.publicUrl`

**Storage bucket decision (Claude's discretion per CONTEXT.md):**
Use a dedicated `inspection-photos` bucket (matches the pattern of `property-images` for property photos — separate bucket by domain). This avoids path collision and simplifies RLS per bucket.

All queryFns use `createClient()` from `#lib/supabase/client` and `handlePostgrestError` from `#lib/postgrest-error-handler`. Remove `apiRequest` import.

**Inspection CRUD mutations:**

`useCreateInspection`:

`useUpdateInspection(id)`:
- `const { data: updated, error } = await supabase.from('inspections').update(dto).eq('id', id).select().single()`
- onSuccess: `queryClient.setQueryData(inspectionQueries.detailQuery(id).queryKey, updated)`

`useCompleteInspection(id)`:
- Per CONTEXT.md: validate all rooms have condition_rating before marking complete
- `mutationFn`: First fetch the inspection to check rooms:
  ```
  const supabase = createClient()
  const { data: rooms, error: roomsError } = await supabase
    .from('inspection_rooms')
    .select('id, condition_rating')
    .eq('inspection_id', id)
  if (roomsError) handlePostgrestError(roomsError, 'inspection_rooms')
  const unassessed = rooms?.filter(r => !r.condition_rating) ?? []
  if (unassessed.length > 0) {
    throw new Error(`All rooms must be assessed before completing. ${unassessed.length} room(s) have no condition rating.`)
  }
  const { data: updated, error } = await supabase
    .from('inspections')
    .update({ status: 'completed', completed_at: new Date().toISOString() })
    .eq('id', id)
    .select()
    .single()
  if (error) handlePostgrestError(error, 'inspections')
  return updated
  ```
- onSuccess: invalidate both detail and list caches

`useSubmitForTenantReview(id)`:
- This is a pure DB status update (no email in this mutation — email would be handled by n8n/DB webhook later in Phase 56):
- `const { data: updated, error } = await supabase.from('inspections').update({ status: 'tenant_reviewing' }).eq('id', id).select().single()`
- onSuccess: invalidate detail + list

`useTenantReview(id)`:
- Per CONTEXT.md: tenant-review mutations that require external services should defer with TODO
- The `TenantReviewInput` includes `tenant_signature_data` — if signature storage is pure DB (no DocuSeal), migrate to PostgREST. If DocuSeal is involved (e.g., generating a signed PDF), defer.
- Based on the inspections.service.ts logic: tenant review stores `tenant_notes` and `tenant_signature_data` in the DB and sets `status = 'finalized'`. This is a pure DB operation (DocuSeal is used for leases, not inspection reviews).
- Migrate to PostgREST: `supabase.from('inspections').update({ ...dto, status: 'finalized', tenant_reviewed_at: new Date().toISOString() }).eq('id', id).select().single()`

`useDeleteInspection`:
- `const { error } = await supabase.from('inspections').delete().eq('id', id)`
- No retention requirement for inspection records

- From base_schema.sql, check FK behavior. If uncertain, delete photos from `inspection_photos` table first then delete the room.
