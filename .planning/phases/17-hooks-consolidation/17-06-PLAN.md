<task type="auto">
  <name>Task 1: Detect and delete dead hooks (zero imports)</name>
  <files>
    src/hooks/api/,
    src/hooks/
  </files>
  <action>
Scan every exported hook function across `src/hooks/` for zero imports. Per CONTEXT.md: "Confirmed dead hooks (zero imports in src/ and tests/) should be deleted entirely." Per user note: "Knip is not reliable enough to put any amount of blind trust into" -- use grep-based verification, not Knip.

**Step 1: Build the hook export inventory.**
For every `.ts` and `.tsx` file in `src/hooks/` (recursively), extract all exported function names:
```bash
grep -rn "export function\|export const\|export default function" src/hooks/ --include="*.ts" --include="*.tsx" | grep -oP '(?:export (?:function|const|default function) )\K\w+'
```

**Step 2: For each exported hook, check import count.**
For each exported function name, grep across `src/` and `tests/` for imports:
```bash
# For each hook_name:
grep -rn "$hook_name" src/ tests/ --include="*.ts" --include="*.tsx" | grep -v "export" | wc -l
```

A hook is "dead" if it has zero non-export references across `src/` and `tests/`.

**Step 3: Delete confirmed dead hooks.**
For each dead hook:
1. If the hook is the only export in its file, delete the entire file
2. If the file has other live exports, remove only the dead hook function
3. After deletion, re-run typecheck to catch any transitive dependencies the grep missed

**Step 4: Verify no breakage.**
```bash
pnpm typecheck
pnpm test:unit -- --run src/hooks/
```

If typecheck fails after a deletion, the hook was NOT dead (indirect import via re-export or type-only usage). Restore it and remove from the dead list.

IMPORTANT: Do NOT delete hooks that are only used in test files -- those are still "alive" for testing purposes. Only delete hooks with truly zero references anywhere.

<task type="auto">
  <name>Task 2: Audit for overlapping hook functionality and resolve</name>
  <files>
    src/hooks/api/,
    src/hooks/
  </files>
  <action>
Scan for overlapping hook functionality across the remaining live hooks. Two hooks "overlap" if they:
- Wrap the same Supabase query/RPC with different names
- Provide the same mutation with different invalidation patterns
- Duplicate the same data transformation logic

**Step 1: Identify potential overlaps.**
Look for hooks that query the same table or call the same RPC:
```bash
# Find all supabase.from() calls in hooks
grep -rn "\.from(" src/hooks/ --include="*.ts" | sort
# Find all supabase.rpc() calls in hooks
grep -rn "\.rpc(" src/hooks/ --include="*.ts" | sort
```

Group by table/RPC name. If two hooks in different files call the same `.from('table')` or `.rpc('function')`, they are overlap candidates.

**Step 2: Evaluate each overlap candidate.**
For each pair, determine if the overlap is:
- **Intentional domain separation** (e.g., owner vs tenant hooks querying same table with different RLS context) -- keep both, document as intentional
- **Actual duplication** (same query, same context, different names) -- merge into one, update all consumers
- **Partial overlap** (similar but not identical queries) -- extract shared logic into the query-key factory if beneficial, or keep separate if the difference is meaningful

Per CONTEXT.md: "Owner vs tenant domain separation is generally intentional -- don't blindly merge across domains."

**Step 3: Resolve actual duplicates.**
For genuine duplicates:
1. Choose the canonical hook (the one with more consumers or better naming)
2. Update all consumers of the duplicate to import the canonical hook
3. Delete the duplicate
4. Run typecheck and tests

**Step 4: Document findings.**
In the SUMMARY, document:
- How many dead hooks were found and deleted (from Task 1)
- How many overlap candidates were found
- Which were intentional (owner/tenant separation)
- Which were resolved (merged/deleted)
- Final hook count vs starting count

After all changes:
```bash
pnpm typecheck && pnpm test:unit && pnpm lint
```
