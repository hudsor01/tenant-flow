# 62-01: Error Handling ‚Äî Eliminate Double-Toast Pattern

## Goal
Error toasts appear exactly once per error. No duplicate toasts anywhere in the app.

## Root Cause
`handlePostgrestError()` shows a toast AND throws. When TanStack Query catches the throw in `onError`, `handleMutationError()` shows ANOTHER toast. Two toasts per error.

## Tasks

### T1: Remove toast from handlePostgrestError
**File**: `apps/frontend/src/lib/postgrest-error-handler.ts`
- Remove `toast.error(...)` call
- Keep Sentry capture and throw behavior
- Add JSDoc clarifying this is throw-only (no user-facing side effects)
- Remove `toast` import and `sonner` dependency

### T2: Audit mutations that use handlePostgrestError WITHOUT handleMutationError in onError
Some mutation hooks call `handlePostgrestError` inside `mutationFn` but have NO `onError` callback. After T1, these would silently fail with no toast. Find them and add `onError: (error) => handleMutationError(error, 'context')`.

**Hooks to check** (all files with handlePostgrestError in hooks/api/):
- use-properties.ts ‚Äî mutations: create, update, delete, deleteImage
- use-unit.ts ‚Äî mutations: create, update, delete
- use-lease.ts ‚Äî mutations: create, update, terminate, renew, delete, and 6 more
- use-tenant.ts ‚Äî mutations: create, update, delete, moveOut, invite, resend, cancel, batch ops
- use-vendor.ts ‚Äî mutations: create, update, delete, assign, unassign
- use-maintenance.ts ‚Äî mutations: create, delete, update
- use-inspections.ts ‚Äî mutations: all 11 inspection mutations
- use-payments.ts ‚Äî mutations: recordManual, export, process, sendReminder, setDefault, delete
- use-payment-methods.ts ‚Äî mutations: delete, setDefault, add
- use-reports.ts ‚Äî mutations: delete, download
- use-billing.ts ‚Äî mutations
- use-tenant-portal.ts ‚Äî mutations
- use-owner-notification-settings.ts ‚Äî mutations
- use-emergency-contact.ts ‚Äî mutations

### T3: Remove duplicate toast calls from individual hooks
Some hooks have `toast.success(...)` in `onSuccess` AND ALSO call `handleMutationSuccess()` which shows its own toast. Check for doubles in success path too.

Scan for hooks that call `toast.success` directly in `onSuccess` ‚Äî these are fine as long as they DON'T also call `handleMutationSuccess`. If they do both, remove the direct `toast` call.

### T4: Verify no double-toast in query error paths
`handlePostgrestError` is also called in `queryFn` functions (not just mutations). After T1, query errors will have no user-facing toast. This is actually correct ‚Äî query errors should show via error boundaries or component error states, not toasts. Verify this is acceptable.

## Acceptance Criteria
- [ ] `handlePostgrestError` never calls `toast`
- [ ] Every mutation that can fail via PostgREST has an `onError` callback with `handleMutationError`
- [ ] No hook has both `toast.error(...)` and `handleMutationError(...)` for the same error
- [ ] `pnpm typecheck` passes
- [ ] `pnpm lint` passes
