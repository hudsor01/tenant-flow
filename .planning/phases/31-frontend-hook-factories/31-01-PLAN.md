    2. Export `useEntityDetail<T extends { id: string }>(config: EntityDetailConfig<T>)` that:
       - Calls `useQueryClient()` only when `config.listQueryKey` is provided (conditional is fine since it is not a conditional hook call -- the hook itself is always called, only the placeholderData function uses queryClient).
       - Actually: ALWAYS call `useQueryClient()` unconditionally (React Compiler requires stable hook calls). Store in variable. Only USE it inside placeholderData when listQueryKey is defined.
       - Returns `useQuery({ ...config.queryOptions, ...(config.listQueryKey ? { placeholderData } : {}) })`.
       - The `placeholderData` function calls `queryClient.getQueriesData<{ data?: T[] } | T[]>({ queryKey: config.listQueryKey })`, iterates results, handles both `{ data: T[] }` (PaginatedResponse) and `T[]` (plain array) shapes, and returns the first item where `entry.id === config.id`.

    Create `src/hooks/create-mutation-callbacks.ts` (per D-01, D-02, FRONT-03):

    1. Define `MutationCallbackConfig<TData>` interface with fields:
       - `invalidate`: `ReadonlyArray<readonly unknown[]>` -- query keys to invalidate on success.
       - `successMessage` (optional): `string` -- toast message on success.
       - `errorContext`: `string` -- context for handleMutationError (e.g., "Create property").
       - `updateDetail` (optional): `(data: TData) => { queryKey: readonly unknown[]; data: TData }` -- Tier 2: set detail cache on success.
       - `removeDetail` (optional): `(data: TData, variables: unknown) => readonly unknown[]` -- remove detail cache entry on success (for deletes).
       - `onSuccessExtra` (optional): `(data: TData) => void` -- custom callback after standard operations.
       - `useSuccessHandler` (optional): `boolean` -- when true, use `handleMutationSuccess(errorContext, successMessage)` instead of `toast.success(successMessage)`. This addresses the handleMutationSuccess vs toast.success divergence (Research Pitfall 3).

    2. Export `createMutationCallbacks<TData, TVariables = unknown>(queryClient: QueryClient, config: MutationCallbackConfig<TData>)` that returns `{ onSuccess, onError }`:
       - `onSuccess(data: TData, variables: TVariables)`:
         a. If `config.updateDetail`, call `queryClient.setQueryData(result.queryKey, result.data)`.
         b. If `config.removeDetail`, call `queryClient.removeQueries({ queryKey: result })`.
         c. For each key in `config.invalidate`, call `queryClient.invalidateQueries({ queryKey: key as QueryKey })`.
         d. If `config.useSuccessHandler` and `config.successMessage`, call `handleMutationSuccess(config.errorContext, config.successMessage)`.
         e. Else if `config.successMessage`, call `toast.success(config.successMessage)`.
         f. Call `config.onSuccessExtra?.(data)`.
       - `onError(error: unknown)`: call `handleMutationError(error, config.errorContext)`.

    3. Export the `MutationCallbackConfig` type so consuming hooks can reference it if needed.

    - `import { useQuery, useQueryClient } from '@tanstack/react-query'`

    Do NOT create `useMutationWithCallbacks` convenience hook yet -- keep it simple. Each consuming hook calls `useQueryClient()` and passes the client to `createMutationCallbacks()`. This matches the existing pattern where every hook already calls `useQueryClient()`.
