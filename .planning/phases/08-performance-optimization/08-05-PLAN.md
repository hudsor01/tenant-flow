<objective>
Eliminate the tenant portal waterfall by sharing tenant ID resolution and parallelizing independent queries within amountDue.

Purpose: Currently, each of the 4 tenant portal hooks independently queries `tenants.select('id').eq('user_id', user.id).single()` -- that's 8x redundant tenant ID resolution across all hooks. The amountDue query in use-tenant-payments.ts has a 5-step sequential waterfall: user -> tenant -> lease -> [connected_account, rent_due] -> existing_payment. By caching the tenant ID and parallelizing independent queries, we reduce round-trips from 5 to 3 (user, then tenant-from-cache, then parallel independent queries).

Output: Shared tenantIdQuery in use-tenant-portal-keys.ts, 4 tenant hooks refactored to use it.
</objective>

**use-tenant-portal-keys.ts:**

Read the file first. Add a shared `tenantIdQuery` that all tenant portal hooks can use:

```typescript
import { queryOptions } from '@tanstack/react-query'
import { createClient } from '#lib/supabase/client'
import { getCachedUser } from '#lib/supabase/get-cached-user'

export const tenantIdQuery = () =>
  queryOptions({
    queryKey: [...tenantPortalKeys.all, 'tenant-id'],
    queryFn: async (): Promise<string | null> => {
      const supabase = createClient()
      const user = await getCachedUser()
      if (!user) throw new Error('Not authenticated')
      const { data } = await supabase
        .from('tenants')
        .select('id')
        .eq('user_id', user.id)
        .single()
      return data?.id ?? null
    },
    staleTime: 10 * 60 * 1000,  // tenant_id is immutable within session
    gcTime: 30 * 60 * 1000,
  })
```

Also export a helper function that can be called inside other queryFns to get the cached tenant ID (avoiding the hook-level dependency):

```typescript
export async function resolveTenantId(): Promise<string | null> {
  const supabase = createClient()
  const user = await getCachedUser()
  if (!user) return null
  const { data } = await supabase
    .from('tenants')
    .select('id')
    .eq('user_id', user.id)
    .single()
  return data?.id ?? null
}
```

The `tenantIdQuery` is for use as a hook (useQuery dependency pattern). The `resolveTenantId` function is for use inside other queryFns where the TanStack Query cache handles deduplication at the query level.

**use-tenant-payments.ts amountDue query (PERF-01):**

Refactor the amountDue queryFn to parallelize independent queries after the lease is fetched:

Current flow (5 sequential steps):
1. getCachedUser() -> user
2. tenants.select('id').eq('user_id', user.id) -> tenantRecord
3. leases.select(...).eq('lease_tenants.tenant_id', tenantRecord.id) -> lease
4. stripe_connected_accounts.select(...) -> chargesEnabled (SEQUENTIAL)
5. rent_due.select(...) + rent_payments check (SEQUENTIAL)

Optimized flow (3 rounds):
1. getCachedUser() -> user
2. resolveTenantId() -> tenantId (cached by TanStack Query)
3. leases.select(...) -> lease
4. PARALLEL: [connected_account, rent_due] (both depend only on lease.owner_user_id and lease.id which are available)
5. rent_payments check (depends on rent_due.id from step 4)

```typescript
// After getting the lease (step 3), parallelize connected account + rent_due:
const [connectedAccountResult, rentDueResult] = await Promise.all([
  supabase.from('stripe_connected_accounts')
    .select('charges_enabled')
    .eq('user_id', leaseRaw!.owner_user_id)
    .maybeSingle(),
  supabase.from('rent_due')
    .select('id, amount, due_date, status')
    .eq('lease_id', lease.id)
    .gte('due_date', today)
    .order('due_date')
    .limit(1)
    .maybeSingle(),
])
```

Replace the inline tenant ID resolution with `resolveTenantId()` (imported from use-tenant-portal-keys.ts). This eliminates the redundant DB call when the tenant portal loads multiple hooks.

Also update the payments() queryFn to use `resolveTenantId()` instead of inline tenant resolution.
