Files: `send-tenant-invitation`, `export-user-data`, `tenant-invitation-accept`
These create a user-scoped client with anon key for auth, then a separate service-role client for DB ops.

For each file:
- Keep: `import { createClient } from '@supabase/supabase-js'` (needed for the anon-key user-scoped client)

- The user-scoped client pattern remains:
```typescript
const supabaseAuth = createClient(env['SUPABASE_URL'], env['SUPABASE_ANON_KEY'], {
  global: { headers: { Authorization: `Bearer ${token}` } }
})
```
HOWEVER, `validateBearerAuth` extracts the token internally and calls `getUser()`. So we need the token BEFORE constructing the user-scoped client. Approach:
1. Construct the user-scoped client WITHOUT the token header (just anon key)
2. Pass it to `validateBearerAuth` which extracts the token and calls `getUser(token)`
3. The `getUser(token)` call works on any client -- it validates the token server-side regardless of the client's own auth header

Actually, re-reading the current code more carefully: the anon-key client is created with `global.headers.Authorization` set to the Bearer token. This is done so that the client's requests are scoped to that user for RLS. But the `getUser(token)` call passes the token explicitly, so it does NOT depend on the client's own auth header.

The simplest approach: create the anon-key client WITHOUT the token header, use it for `validateBearerAuth`, then the service-role client for DB operations. The anon-key-with-token-header client was only used for the `getUser` call anyway.

Wait -- re-reading `send-tenant-invitation`:
```typescript
const supabaseAuth = createClient(env['SUPABASE_URL'], env['SUPABASE_ANON_KEY'], {
  global: { headers: { Authorization: `Bearer ${token}` } }
})
const { data: { user }, error: authError } = await supabaseAuth.auth.getUser(token)
```
The `supabaseAuth` client is ONLY used for the single `getUser(token)` call. After that, a separate service-role client is created for all DB operations. Same pattern in `export-user-data` and `tenant-invitation-accept`.

So the approach is:
1. Create the anon-key client (plain, no token header): `const supabaseAuth = createClient(env['SUPABASE_URL'], env['SUPABASE_ANON_KEY'])`
2. Pass it to `validateBearerAuth(req, supabaseAuth)` -- this extracts Bearer token and calls `supabaseAuth.auth.getUser(token)`
3. Create admin client for DB: `const supabase = createAdminClient(env['SUPABASE_URL'], env['SUPABASE_SERVICE_ROLE_KEY'])`
4. Remove the old `global: { headers: ... }` option from the anon client -- it was unnecessary
