# Roadmap: TenantFlow v2.2 — Landlord-First Positioning

**Defined:** 2026-04-20
**Target ship:** 2026-05-11 (3 weeks, 3 phases)
**Pauses:** v2.0 Phase 46 (premium reports gate) — stays paused; won't ship until dashboard + marketing tell the right story.

## Why This Milestone Exists

The landlord-only pivot (PR #596) ripped out rent facilitation, the tenant portal, autopay, and Stripe Connect — but the codebase's UI copy, component names, and marketing language still describe the old product. A user who signs up today sees "Invite tenant" buttons that create landlord-managed records (the button lies), reads marketing copy about "rent collection" (a feature we no longer have), and lands on a pricing page that implies features we removed.

This milestone realigns what the product *says* with what it *does*.

**Success criterion for the milestone:**
- Zero occurrences of "invite tenant", "rent collection", "autopay", "tenant portal", "pay rent" anywhere in user-facing copy (UI labels, marketing pages, email templates, error messages)
- Homepage + pricing + features pages describe the actual product promise: landlord operations without tenant auth accounts
- New signup flow lands on an empty-state dashboard that explains what to do next
- Trial banner visible on every authenticated page showing days remaining from `users.trial_ends_at`

## Phases (strict ordering — 53 depends on 52, 54 can wait for feedback)

- [x] **Phase 52: Dashboard copy + component rename** (shipped 2026-04-20, PR #609)
- [ ] **Phase 53: Public-facing value-prop reframe** (1 week)
- [ ] **Phase 54: Dashboard UX polish** (2 weeks)

## Phase 52: Dashboard Copy + Component Rename

**Goal:** Every label/button/component name in the authenticated app matches the landlord-only product model. A landlord who already signed up stops seeing stale terminology.

**Requirements:**
- **REQ-52-01**: Rename `src/components/tenants/invite-tenant-*` files + their React components to `add-tenant-*`. Update every import. Matching tests renamed.
- **REQ-52-02**: UI copy pass — grep every "invite" / "Invite" in `src/app/(owner)/**` and `src/components/**`, replace with "add" / "Add" where semantically correct. "Invite" only survives in contexts that are actually invitations (if any — none expected post-pivot).
- **REQ-52-03**: Empty-state rewrites — `/tenants` "No tenants yet" copy explains: "Tenants are records you keep for your own tracking. They don't log in." `/leases` empty-state: "Create a lease between one of your properties and a tenant record." Etc.
- **REQ-52-04**: Delete genuinely-dead invitation-flow code: `useResendInvitationMutation`, `useCancelInvitationMutation`, any invitation-accept route helpers still lingering.
- **REQ-52-05**: Trial banner component — reads `users.trial_ends_at`, shows "X days left in trial — upgrade to keep access" on authenticated pages. Links to `/billing/plans?source=trial_banner`.
- **REQ-52-06**: Expand `marketing-copy-landlord-only.test.ts` pattern to cover `src/components/**` — fails if any forbidden phrase re-appears.

**Battle-proven criterion:** run `rg -i "invite tenant|invite a tenant|rent collection|autopay|tenant portal|pay rent"` in `src/app/(owner)/**` and `src/components/**` — zero matches. Live `/dashboard` for a trialing owner shows the banner with a real day count.

## Phase 53: Public-Facing Value-Prop Reframe

**Goal:** A first-time visitor to tenantflow.app understands what the product does (landlord operations tool, not a payments platform) and why they'd pay for it.

**Depends on:** Phase 52 must ship first — can't market "add tenants in seconds" while the app button still says Invite.

**Requirements:**
- **REQ-53-01**: Homepage hero rewrite — new positioning: "The operations tool for small landlords. Track properties, tenants, leases, and maintenance — without the spreadsheet." Replace rent-collection-centric headline.
- **REQ-53-02**: Features page — drop sections describing removed features (rent collection, autopay, tenant portal). Replace with features the product actually has: property management, lease tracking, maintenance requests, e-signature via DocuSeal (gated), financial reports, tenant screening-readiness (read-only records).
- **REQ-53-03**: Pricing page — each tier lists only what it actually unlocks. Revise the "What's included" lists. Add a "Free 14-day trial, no credit card required" banner (true post-Phase-51).
- **REQ-53-04**: FAQ — remove questions about removed features. Add questions about the landlord-only model ("Do my tenants create accounts?" — "No, they don't. Tenants are records you manage.")
- **REQ-53-05**: About page — any copy about the old product direction, rewrite or remove.
- **REQ-53-06**: SEO — titles / meta descriptions / OG tags on every marketing page reflect new positioning. Update `src/app/sitemap.ts` / `robots.ts` if any removed routes are still listed.
- **REQ-53-07**: Blog — audit every post for references to removed features. Either rewrite, add an "outdated" banner, or unpublish (keep URLs for SEO, add redirect or noindex).
- **REQ-53-08**: Auth-email templates (`_shared/auth-email-templates.ts`) — confirm/signup/recovery/invite/magiclink copy all landlord-only; drop any mention of rent or tenant accounts. (Separate from the already-deleted drip templates.)
- **REQ-53-09**: Footer links — remove any to removed features. Add links to support + help that match the new positioning.

**Battle-proven criterion:** Marketing regression test (REQ-52-06 extended to all public pages) passes. Manual review by one external reader ("does this tell you what the product does?") returns yes.

## Phase 54: Dashboard UX Polish

**Goal:** New signups get guided to a productive first session, and active landlords spend less time hunting for things.

**Depends on:** Phases 52 + 53 shipped. Ideally informed by 1-2 weeks of real trial-user feedback.

**Requirements:**
- **REQ-54-01**: Dashboard landing — for zero-property users, replace the data grid with a guided "Add your first property" CTA. Show sample screenshots/mockup of what the dashboard will look like once populated. For users with data, keep current stats.
- **REQ-54-02**: Sample-data seeder — optional one-click "Load sample data" button for users who want to explore the product without committing. Seeds a property + unit + tenant + lease + maintenance request, all flagged with `is_sample = true`. "Clear sample data" button removes it.
- **REQ-54-03**: Settings IA — remove settings pages for removed features (Stripe Connect onboarding, autopay preferences, payment methods). Consolidate remaining settings into Account / Billing / Notifications.
- **REQ-54-04**: Sidebar reorg — current order grew organically. Reorder by frequency-of-use (Dashboard → Properties → Tenants → Leases → Maintenance → Reports → Settings). Collapse rarely-used items into a "More" submenu.
- **REQ-54-05**: Empty-state illustrations — simple SVG icons or Lucide icons on each empty state, not just text.
- **REQ-54-06**: Toast + loading-state consistency audit — same skeleton pattern everywhere; same success-toast tone; same error-toast format.

**Kill criteria:** if after shipping 52 + 53, users aren't asking for any of the 54 items, cut the phase and move to whatever they ARE asking for.

## Out of Scope

- Any new product features (stays deferred per v2.0 roadmap)
- Visual brand refresh (colors, fonts, logo) — not a scope blocker
- A/B testing infrastructure — premature at <20 users
- Internationalization — US-English only
- Accessibility overhaul — keep existing a11y level; major a11y audit is a separate milestone

## Sequencing Rules

1. Phase 52 ships first. Copy must match reality BEFORE driving traffic.
2. Phase 53 ships after Phase 52's regression test is green. Marketing can't promise what the app doesn't say.
3. Phase 54 is gated on real-user feedback. If no specific UX complaints arrive within 2 weeks of 53, treat 54 as "designed but not built" and defer.
4. Each phase ships as its own PR. No bundling. Regression test always green.
