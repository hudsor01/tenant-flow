## SEO-01 ‚Äî Properties page N+1 query fan-out
- **Finding:** src/app/(owner)/properties/page.tsx:70 (medium) ‚Äî page fires `useQueries` with `unitQueries.listByProperty(p.id)` per property (line 70) plus a second `useQueries` with `propertyQueries.images(p.id)` per property (line 90), up to 1+2N=101 waterfalled round-trips per page load.
- **Root cause:** No consolidated query factory exists for "property list + units + cover image", so the page composes three separate factories client-side. `propertyQueries.withUnits()` (property-keys.ts:89) proves the embedded-select pattern but is itself deficient (unbounded `select("*, units(*)")`, no images, no count/range, no inactive-unit filter) and is consumed only by the test-only `usePropertiesWithUnits` hook.
- **Fix:** Replace `propertyQueries.withUnits()` in `src/hooks/api/query-keys/property-keys.ts` with a consolidated `propertyQueries.listWithDetails(filters?: PropertyFilters)` that issues ONE PostgREST request:
  - `.select(`${PROPERTY_SELECT_COLUMNS}, units(<UNIT_SELECT_COLUMNS>), property_images(image_url, display_order)`, { count: "exact" })`
  - parent filters identical to `list()` (`.neq("status","inactive")` unless status given, property_type, sanitized search, `.range(offset, offset+limit-1)` with default limit 50)
  - embedded filters/order: `.neq("units.status","inactive")`, `.order("unit_number", { referencedTable: "units", ascending: true })`, `.order("display_order", { referencedTable: "property_images", ascending: true })`, `.limit(1, { referencedTable: "property_images" })` (PostgREST applies referenced-table order/limit per parent row)
  - typed mapper (no `as unknown as`) returning `PaginatedResponse<PropertyWithUnits & { cover_image_url: string | null }>` ‚Äî reuse `PropertyWithUnits` from `src/types/relations.ts:94` and add the cover extension type there per Type Lookup Order; the mapper resolves relative `image_url` paths through `supabase.storage.from("property-images").getPublicUrl()` (synchronous, no extra round-trip ‚Äî same logic as `propertyQueries.images`).

  `src/app/(owner)/properties/page.tsx`: replace the `propertyQueries.list()` useQuery + both `useQueries` blocks + `unitsMap`/`imagesMap` with one `useQuery(propertyQueries.listWithDetails())`; `transformToPropertyItem(p, p.units, p.cover_image_url ?? undefined)` (signature in property-transforms.ts already takes `(property, units, imageUrl)` ‚Äî unchanged). Drop the now-unused `unitQueries` import and `useQueries` import. Update `usePropertiesWithUnits` in `src/hooks/api/use-properties.ts:62` (and its test) to wrap the new factory. Because the per-property `propertyQueries.images(id)` key no longer feeds the list view, the image mutations that today invalidate only that key (`src/components/properties/property-image-dropzone.tsx:113`, `src/components/properties/property-form-upload.ts:98`) must additionally invalidate `propertyQueries.lists()` so the list thumbnail refreshes after upload/delete.
- **Why it fixes it:** Verifier counted 1+2N requests all gated behind `propertyQueries.list()`; the consolidated factory reduces the page to exactly 1 request with no waterfall, keeps `{ count: 'exact' }` pagination, and preserves the inactive-unit filter `listByProperty` applied.
- **Risks / interactions:** Embedded per-parent `limit`/`order` requires the `referencedTable` option form (supported by supabase-js v2 / PostgREST 12 ‚Äî prod runs it); verify in the PR with a live query. `unitQueries.listByProperty` keeps its other consumers ‚Äî do not delete. Phase 39 (DATA) and Phase 41 (COMP) run before 48 and may touch property-keys.ts / properties page ‚Äî implement on top of their landed state. Mutation-invalidation rule (related keys + `ownerDashboardKeys.all`) already satisfied by the delete mutation on the page; the new invalidations extend the image mutations only.
- **Files touched:** src/hooks/api/query-keys/property-keys.ts, src/app/(owner)/properties/page.tsx, src/hooks/api/use-properties.ts, src/types/relations.ts, src/components/properties/property-image-dropzone.tsx, src/components/properties/property-form-upload.ts, src/hooks/api/__tests__/use-properties.test.tsx
- **Decision:** Chosen: single embedded-select query (1 round-trip, canonical factory). Alternative: keep `list()` and add two batched `.in("property_id", ids)` queries (3 round-trips, smaller diff but still a waterfall and two more factories to maintain) ‚Äî rejected as it leaves the request waterfall in place.

## SEO-02 ‚Äî Markdown pipeline ships in the client bundle for every blog post
- **Finding:** src/app/blog/[slug]/markdown-content.tsx:18 (medium) ‚Äî react-markdown + remark-gfm + rehype-raw (parse5) + rehype-sanitize cross the client boundary because `"use client"` blog-post-page.tsx statically imports MarkdownContent (line 19).
- **Root cause:** The article body is rendered *inside* the client component instead of being rendered by the server `page.tsx` and passed down as a slot. Two couplings force this: (1) blog-post-page imports `MarkdownContent` to render the split halves, and (2) blog-post-page imports `headingId` from markdown-content.tsx, so even the ToC pulls the whole module (and its markdown deps) into the client graph.
- **Fix:**
  1. Extract `headingId` into a new dependency-free module `src/app/blog/[slug]/heading-id.ts`; import it from there in both markdown-content.tsx and blog-post-page.tsx (kills coupling 2).
  2. Move `splitContentForCta` + `LEAD_MAGNETS` + the CTA-choice logic out of blog-post-page into the server `page.tsx` (or a small server-only sibling module). `page.tsx` renders the body server-side: `<MarkdownContent content={firstHalf} /> + (LeadMagnetCta | BlogInlineCta when secondHalf) + <MarkdownContent content={secondHalf} />` and passes the result to BlogPostPage as a new `articleBody: ReactNode` prop (React serializes server-rendered trees across the client boundary; client components like LeadMagnetCta render normally inside it).
  3. blog-post-page.tsx drops the `MarkdownContent`/`LeadMagnetCta`/`BlogInlineCta` imports and renders `{articleBody}` inside the existing prose div. `extractToc` stays client-side (pure regex over `post.content`, no markdown dependency) using `headingId` from the new module. Update markdown-content.tsx's header comment ‚Äî it becomes a true Server Component.
  4. Update `src/app/blog/[slug]/page.test.tsx` to pass an `articleBody` prop (tests currently render BlogPostPage with `post` props directly).
- **Why it fixes it:** The verifier's defect is the static import chain `"use client"` ‚Üí MarkdownContent; after the change no client module imports markdown-content.tsx, so react-markdown and the rehype/remark chain stay server-only while the article body still lands in the initial HTML (the SSR/SEO win the file's own comment defends). This is the exact "server renders it and passes it as a slot" remedy the verifier names.
- **Risks / interactions:** MUST NOT introduce `cookies()`/`headers()` anywhere in the new server path ‚Äî `/blog/[slug]` must stay `‚óè` SSG (`dynamicParams=false` 404 gate; guarded by seo-smoke "/blog/bogus returns real HTTP 404"). `BlogPostProps` changes shape ‚Äî update the type in blog-post-page.tsx and every test constructing it. Phase 45 (CONTENT) runs before 48 and may touch blog files; apply on the landed state. Do NOT swap to `next/dynamic({ ssr: false })` ‚Äî that was deliberately removed to restore SSR (see file header).
- **Files touched:** src/app/blog/[slug]/page.tsx, src/app/blog/[slug]/blog-post-page.tsx, src/app/blog/[slug]/markdown-content.tsx, src/app/blog/[slug]/heading-id.ts (new), src/app/blog/[slug]/page.test.tsx
- **Decision:** Chosen: server-rendered slot (keeps SSR HTML AND removes bundle cost). Alternative: `next/dynamic(..., { ssr: false })` per the CLAUDE.md heavy-lib rule ‚Äî rejected because it strips the article body from initial HTML, regressing the documented SEO win.

- **Finding:** src/app/sitemap.ts:153 (medium) ‚Äî `createClient()` from `#lib/supabase/server` calls `cookies()` (server.ts:12), a Dynamic API that opts the route out of ISR; `revalidate = 86400` (line 7) is dead and prod serves `x-vercel-cache: MISS` on every fetch.

- **Files touched:** src/app/sitemap.ts, src/lib/blog/blog-queries.ts
