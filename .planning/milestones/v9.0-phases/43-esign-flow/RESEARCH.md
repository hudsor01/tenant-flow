## SIGN-01 — Tenant retention copy of the executed lease
- **Finding:** src/components/leases/sign-lease-form.tsx:150 (medium) — After signing, the tenant permanently loses all access to the lease: the used token 404s the `document` action, the success card only says "close this page", and nothing emails the tenant a copy.
- **Root cause:** The e-sign flow has no post-execution delivery path for the tenant at all. The token is single-use (`sign_lease_with_token` sets `used_at`), the `document` action requires `row.valid` (sign-lease-token/index.ts:147), `finalizeSignedLease` only renders + uploads to the owner-scoped `tenant-documents` bucket (lease-signing.ts:251-291), and the only tenant-facing emails (lease-signature `send`/`resend`) are pre-sign. Tenants have no accounts, so once the token is consumed there is no channel left.
- **Fix:** Email the tenant the executed signed PDF on the first successful finalize, idempotently:
  1. `supabase/functions/_shared/resend.ts` — extend `SendEmailParams` with optional `attachments?: Array<{ filename: string; content: string }>` (base64 content) and pass through to the Resend API body (`body.attachments`). Add a chunked Uint8Array→base64 encoder (no `String.fromCharCode(...spread)` on large arrays).
  2. New migration (shared with SIGN-03/04): `alter table public.leases add column signed_lease_emailed_at timestamptz;` — write-once send tracking so retries (SIGN-02) never double-email. Column is NOT in the SEC-05 `reject_signature_audit_tampering` list, so the tamper guard is untouched. Regen `src/types/supabase.ts` via `bun run db:types`.
  3. `supabase/functions/_shared/lease-signing.ts` — restructure `finalizeSignedLease` into two idempotent steps: (a) ensure-PDF (current render → upload → pointer write, unchanged guards); (b) ensure-email: if `signed_document_path` is set and `signed_lease_emailed_at is null`, atomically claim it (`update leases set signed_lease_emailed_at = now() ... .is('signed_lease_emailed_at', null).select('id')` — only the claimer proceeds), then `sendEmail` to `record.tenantEmail` with the signed PDF attached (`lease-<id>-signed.pdf`, bytes reused from the just-rendered buffer, or downloaded from `tenant-documents` on the retry path), subject "Your signed lease agreement", `wrapEmailLayout` body with `escapeHtml`'d values, `idempotencyKey: "signed-lease-email-<leaseId>"`. On send failure: reset the claim to null + `captureWebhookError`, so a later finalize retry re-attempts.
  4. `src/components/leases/sign-lease-form.tsx` — success-card copy: bothSigned → "All parties have signed. This lease is now active. A copy of the signed lease is being emailed to you."; tenant-first → merged with SIGN-03 copy ("The landlord has been notified… you'll receive an emailed copy of the fully signed lease once they countersign."). Update `src/components/leases/__tests__/sign-lease-form.test.tsx` copy assertions.
- **Why it fixes it:** The verifier confirmed the consumed token can never re-serve the PDF and no email exists post-sign. A durable emailed executed copy removes any dependence on the consumed/expiring token, satisfies ESIGN record-retention for the tenant, and the success card now tells the tenant what to expect instead of dead-ending.
- **Risks / interactions:** Resend total-message cap is 40MB — pdf-lib lease PDFs are tens of KB, safe. Concurrent finalizes are protected by the atomic claim + Resend idempotency key. Both edge functions change → owner-run deploy (`bun scripts/deploy-edge-functions.ts`, CLI-401 gotcha). Interacts with SIGN-02: the finalize-retry action is what heals a failed email (claim was reset to null). Migration applied via MCP must be filename-reconciled afterwards (migration-mcp-prod-drift memory).
- **Files touched:** supabase/functions/_shared/resend.ts, supabase/functions/_shared/lease-signing.ts, supabase/migrations/20260712000000_esign_notify_reason_order_email_tracking.sql (new, shared), src/components/leases/sign-lease-form.tsx, src/types/supabase.ts (regen), src/components/leases/__tests__/sign-lease-form.test.tsx, supabase/functions/tests/lease-signing-test.ts
- **Decision:** Chosen: email the executed PDF as an attachment. Alternative A — allow the `document` action for a used-but-authentic token (serving the signed variant): rejected because it keeps a PII-serving endpoint alive on a consumed credential, still dies at token expiry (14 days), and is weaker retention than an inbox copy. Alternative B — gate the email purely on the first pointer write with no `signed_lease_emailed_at` column (no migration): rejected because a failed email after a successful pointer write would then be permanently unrecoverable.

## SIGN-02 — Failed finalize has no retry path
- **Finding:** supabase/functions/_shared/lease-signing.ts:274 (medium) — A render/upload failure at finalize leaves `signed_document_path` null forever; the frontend poll only re-reads the DB and the "Finalizing signed document…" button never resolves.
- **Root cause:** `finalizeSignedLease` is invoked exactly once, fire-and-forget, by whichever sign action observes `both_signed` (lease-signature/index.ts:519, sign-lease-token/index.ts:233). On failure it logs to Sentry and returns; no other code path ever re-runs the render/upload — `leaseQueries.signedDocument` (lease-keys.ts:166-208) and the button's manual re-check (download-signed-lease-button.tsx:85) only re-read `signed_document_path`. The comment "The PDF can be regenerated later" has no implementation.
- **Fix:** Add an owner-triggerable, idempotent `finalize` action and wire the frontend to invoke it:
  1. `supabase/functions/lease-signature/index.ts` — new `finalize` action after `loadOwnedLease` (ownership already enforced): rate-limit per owner (`prefix: "lease-esign-finalize"`, identifier `user.id`, 30/min — it renders a PDF), call `finalizeSignedLease(supabase, leaseId)` (internally no-ops unless both signed / already finalized), re-read `signed_document_path`, return `{ success: true, document_ready: boolean }`.
  2. `src/hooks/api/mutation-keys.ts` — add `leases.finalizeSignature` key.
  3. `src/hooks/api/query-keys/lease-mutation-options.ts` — add `finalizeSignature: () => mutationOptions(...)` calling `callLeaseSignatureEdgeFunction("finalize", { leaseId })`.
  4. `src/hooks/api/use-lease-signature-mutations.ts` — add `useFinalizeSignedLeaseMutation`: onSuccess invalidates `leaseQueries.signedDocument(leaseId)` + `leaseQueries.detail(leaseId)`; onError `handleMutationError`.
  5. `src/components/leases/download-signed-lease-button.tsx` — in the `finalizing` branch: auto-fire the finalize mutation once per mount when `finalizing` is first observed (ref-guarded), and make the button click fire finalize-then-refetch instead of bare `refetch()`. Update `download-signed-lease-button.test.tsx`.
- **Why it fixes it:** The verifier confirmed the only callers are at signature time and every frontend path merely re-reads the pointer. The PDF is deterministic from lease data (finding's own premise), so an idempotent owner-side re-render + upload converts a transient Storage/render outage into a self-healing state: the next lease-detail view (or button click) regenerates it.
- **Risks / interactions:** Concurrent finalize calls both render + `upsert` the same deterministic path — benign; the pointer write stays guarded by `.is("signed_document_path", null)`. Must land WITH SIGN-01's claim-guarded email step so retries don't double-email. Edge deploy owner-run. Query keys must come from existing factories (zero-tolerance rule 9) — all invalidations above use `leaseQueries`.
- **Files touched:** supabase/functions/lease-signature/index.ts, src/hooks/api/mutation-keys.ts, src/hooks/api/query-keys/lease-mutation-options.ts, src/hooks/api/use-lease-signature-mutations.ts, src/components/leases/download-signed-lease-button.tsx, src/components/leases/__tests__/download-signed-lease-button.test.tsx
- **Decision:** Chosen: on-demand owner-triggered finalize with frontend auto-fire. Alternative — server-side scheduled retry (pg_cron sweep for active leases with null pointer): rejected because pg_cron cannot render PDFs (Deno-only renderer) and a cron→edge-function invoke queue is disproportionate for a deterministic re-render that auto-fires the moment anyone views the lease.

## SIGN-03 — No owner notification when the tenant signs first
- **Finding:** supabase/functions/sign-lease-token/index.ts:232 (medium) — Tenant-signs-first produces no owner email and no notifications row; the flow silently stalls in `pending_signature` and the tenant success card even claims the landlord will be notified.
- **Root cause:** In `sign_lease_with_token` (migration 20260618200203 lines 93-106) the notification insert lives only inside `if v_lease.owner_signed_at is not null` (the fully-signed branch); there is no `else`. The edge function likewise only acts (`finalizeSignedLease`) when `result.both_signed`. Tenant-first is a fully reachable path because `send` does not require an owner signature.
- **Fix:** Notify on the tenant-signature event itself, both in-app (atomic, in the RPC) and by email (best-effort, in the edge function):
  1. New migration (shared file) — re-define `sign_lease_with_token` (verbatim copy of the 20260618200203 body) adding an `else` branch to the `owner_signed_at` check: `insert into public.notifications (user_id, title, message, notification_type, entity_type, entity_id) values (v_lease.owner_user_id, 'Tenant signed the lease', 'The tenant has signed. Sign as owner to activate the lease.', 'lease', 'lease', v_lease.id);` (same shape as the existing fully-signed insert).
  2. `supabase/functions/sign-lease-token/index.ts` — in the `sign` action, when `result.success && !result.both_signed && result.lease_id`: `fetchLeaseSigningData` → best-effort `sendEmail` to `record.ownerEmail` (guarded by `EMAIL_RE`), subject "Action needed: countersign your lease", `wrapEmailLayout` + `escapeHtml` body with a button linking `${appUrl}/leases/${result.lease_id}`. Add `NEXT_PUBLIC_APP_URL` to this function's `validateEnv` required list (already set project-wide — lease-signature requires it) and import `wrapEmailLayout`/`escapeHtml`/`sendEmail`.
  3. `src/components/leases/sign-lease-form.tsx:165` — fix the false copy: "Thank you. The landlord has been notified and will countersign to activate the lease. You'll receive an emailed copy of the fully signed lease once they do." (single edit merged with SIGN-01's copy change).
- **Why it fixes it:** The verifier confirmed the notification insert is unreachable on the tenant-first path and no other trigger/migration inserts one. The RPC `else` branch makes the in-app notification atomic with the signature write; the email closes the loop promptly so a lease can't sit unsigned past its start date; the card copy becomes truthful.
- **Risks / interactions:** Email failure is logged-only (in-app notification still lands — acceptable degradation). Migration re-defines the same function SIGN-04's change ships beside — one migration file re-defines both RPCs to avoid two prod applies. MCP apply → reconcile filename (prod-drift memory). Edge deploy owner-run. The notifications insert reuses the proven `notification_type = 'lease'` value (no new CHECK exposure).
- **Files touched:** supabase/migrations/20260712000000_esign_notify_reason_order_email_tracking.sql (shared), supabase/functions/sign-lease-token/index.ts, src/components/leases/sign-lease-form.tsx, src/components/leases/__tests__/sign-lease-form.test.tsx

## SIGN-04 — "Already signed / Lease is active" cards unreachable
- **Finding:** src/app/sign/[token]/sign-context.ts:79 (low) — A tenant revisiting their link after signing always gets the warning-styled "Signing link unavailable / already been used" card; the friendly completed-state cards can never render.
- **Root cause:** `get_lease_signing_context`'s reason cascade (20260618200203 lines 177-186) evaluates token-consumption state (`revoked_token` → `used_token` → `expired_token`) before lease-completion state (`tenant_already_signed` / `lease_active`), and signing always consumes the token (plus send/resend revoke all other live tokens) — so the only token a legitimate signer can revisit with reports `used_token`. `isCompletedState` correctly excludes `used_token` because that reason is ambiguous (it also covers cancelled-lease and tenant-changed cases).
- **Fix:** In the same shared migration, re-define `get_lease_signing_context` with the completed-state reasons hoisted to the top of the cascade, gated on the token belonging to the current tenant email (`v_email_match`):
  ```sql
  case
    when v_email_match and v_lease.tenant_signed_at is not null then 'tenant_already_signed'
    when v_email_match and v_lease.lease_status = 'active'      then 'lease_active'
    when v_token.revoked_at is not null                          then 'revoked_token'
    when v_token.used_at is not null                             then 'used_token'
    when v_token.expires_at <= now()                             then 'expired_token'
    when v_lease.lease_status <> 'pending_signature'             then 'lease_not_pending'
    when not v_email_match                                       then 'tenant_changed'
    else null
  end
  ```
  The `valid` boolean expression is unchanged (so the `document` action's PII gate is untouched). No frontend change required — page.tsx:91 + `isCompletedState` already route `tenant_already_signed`/`lease_active` to the CheckCircle2 success card.
- **Why it fixes it:** The verifier pinned the defect to cascade order. With signed/active checked first for the authentic tenant's token, the post-sign revisit yields `tenant_already_signed` → the success card. The `v_email_match` gate prevents a rebound/stale token holder (tenant-changed case) from being falsely told "You have already signed"; those still fall through to the token-state reasons.
- **Risks / interactions:** Revoked-but-authentic tokens (superseded by resend) on a now-signed lease also show "Already signed" — an intentional UX improvement, no data exposure (reason-only; lease details are still withheld for invalid tokens at sign-lease-token/index.ts:102-107). Cancelled-lease + used-token revisits still show `used_token` (accurate). Pure-RPC change; `sign_lease_with_token`'s own error cascade is untouched (mid-sign errors like `used_token` on a double-submit remain correct).
- **Files touched:** supabase/migrations/20260712000000_esign_notify_reason_order_email_tracking.sql (shared)
- **Decision:** Chosen: RPC cascade reorder. Alternative — frontend-only: treat `used_token` as a completed state in `isCompletedState`: rejected because `used_token` is ambiguous (cancelled lease, tenant changed, someone else's consumed link) and the card would falsely confirm a signature the DB may not hold.

## SIGN-06 — `context` action has no per-IP rate ceiling
- **Finding:** supabase/functions/sign-lease-token/index.ts:83 (low) — `context` keys its rate limit on `tokenHash`, so every distinct token opens a fresh 60/min bucket; a single IP can probe tokens / drive the multi-join RPC + Upstash at unbounded aggregate rate on this public (verify_jwt=false) endpoint.
- **Root cause:** The per-token keying is a deliberate tradeoff (comments at lines 75-78: legitimate context fetches come from the Next.js server's shared egress IP, so a pure per-IP bucket would let one tenant exhaust another's) — but it was implemented as a *replacement* for the IP key rather than a *layer on top*, leaving zero aggregate protection (`rate-limit.ts:122`: `options.identifier ?? getClientIp(req)` fully replaces the IP).
- **Fix:** Layer a coarse per-IP ceiling in front of the existing per-token bucket in the `context` branch of `supabase/functions/sign-lease-token/index.ts`:
  ```ts
  // Coarse aggregate ceiling per IP (token-rotation probing), generous enough
  // for the shared Next.js egress; the per-token 60/min below still isolates
  // legitimate sessions from each other.
  const ipLimited = await rateLimit(req, {
    maxRequests: 300, windowMs: 60_000, prefix: "lease-context-ip",
  });
  if (ipLimited) return ipLimited;
  ```
  then the existing per-token limiter unchanged. Run sequentially (IP first) so an IP-limited request never charges token buckets. Update the explanatory comment to describe the two-layer scheme.
- **Why it fixes it:** The verifier confirmed the identifier fully replaces the IP key. With the layered check, rotating random tokens no longer buys unlimited RPC/Upstash invocations — one IP caps at 300/min aggregate — while the per-token bucket still prevents the shared Vercel egress from letting one tenant starve another (300/min supports ~300 signing-page SSR loads/min per egress IP, far above real traffic). This matches the project rule (per-IP limits on unauthenticated functions) without regressing the documented shared-egress concern.
- **Risks / interactions:** Adds one extra Upstash round-trip per context call (fail-open on Upstash errors, unchanged semantics). Headerless callers share the `'unknown'` bucket — that is the existing documented aggregate-fallback behavior in rate-limit.ts. If Vercel concentrates egress harder than expected, the 300/min constant is a one-line tune. Edge deploy owner-run (same deploy batch as SIGN-01/02/03 changes).
- **Files touched:** supabase/functions/sign-lease-token/index.ts
- **Decision:** Chosen: per-IP coarse layer (300/min) on top of the per-token bucket. Alternative — a single global aggregate bucket for the action (identifier constant): rejected, it lets one attacker IP exhaust the ceiling for ALL legitimate signing-page loads (self-DoS on tenants), whereas per-IP confines the damage to the attacker.
