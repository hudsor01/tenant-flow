## PUBUX-01 — Featured pricing card ignores requiresEmailConfirmation
- **Finding:** src/components/pricing/pricing-card-featured.tsx:269 (high) — `onComplete` destructures only `{ email, tenant_id }`, drops `requiresEmailConfirmation`, and unconditionally fires `subscriptionMutation.mutateAsync`, which 401s (no Bearer) when signup produced no session.
- **Root cause:** The `OwnerSubscribeDialog` contract (owner-subscribe-dialog.tsx:22-26) deliberately carries `requiresEmailConfirmation` so callers can branch, but the handler is a **duplicated inline closure** in each pricing card, and both copies were written before the flag existed. There is no single shared completion handler, so the branch was never added anywhere. When Supabase email confirmations are on, `signUp` returns no session, `signInWithPassword` fails with `email_not_confirmed` (owner-subscribe-dialog.tsx:57-74), `createCheckoutSession` (src/lib/stripe/stripe-client.ts:44-55) omits the `Authorization` header, and `stripe-checkout`'s `validateBearerAuth` rejects → mutation `onError` toasts "Failed to start checkout", the rejection propagates into the dialog's `catch`, and it toasts "Sign up failed" — for a signup that succeeded.
- **Fix (class-wide, covers PUBUX-06):** Extract ONE shared completion handler and use it in both cards.
  1. New file `src/components/pricing/subscribe-complete.ts` exporting:
     ```ts
     export async function completeSubscribeSignup(
       payload: { email: string; tenant_id?: string; requiresEmailConfirmation?: boolean },
       opts: {
         startCheckout: (overrides: { customerEmail: string; tenant_id?: string }) => Promise<unknown>;
         closeDialog: () => void;
       },
     ): Promise<void> {
       if (payload.requiresEmailConfirmation) {
         opts.closeDialog();
         return; // dialog toasts "Check your email to confirm your account before first sign in."
       }
       await opts.startCheckout({
         customerEmail: payload.email,
         ...(payload.tenant_id && { tenant_id: payload.tenant_id }),
       });
       opts.closeDialog();
     }
     ```
  2. `pricing-card-featured.tsx:269-275` and `pricing-card-standard.tsx:292-298`: replace both inline `onComplete` closures with `onComplete={(payload) => completeSubscribeSignup(payload, { startCheckout: (o) => subscriptionMutation.mutateAsync(o), closeDialog: () => setSubscribeDialogOpen(false) })}`.
  3. Unit test `src/components/pricing/__tests__/subscribe-complete.test.ts`: (a) `requiresEmailConfirmation: true` → `startCheckout` NOT called, `closeDialog` called; (b) `false`/absent → `startCheckout` called with `customerEmail` (+ `tenant_id` only when present), then `closeDialog`; (c) `startCheckout` rejection propagates (dialog catch still owns real failures).
- **Why it fixes it:** The verifier evidence is precisely "no session → no Authorization header → 401 → two error toasts despite a successful signup". Branching on the flag means the confirmation-required path never calls `createCheckoutSession` at all; the user gets exactly one truthful toast (the dialog's own "Account created / Check your email…", owner-subscribe-dialog.tsx:80-84) and the dialog closes. The success path is byte-identical to today's behavior. Extracting one handler eliminates the sibling-drift root cause (v8.0 FORMFIX-08 lesson: fix the class, not one instance per cycle).
- **Risks / interactions:** No behavior change for the sessionful path. The confirmation-required user must return via the email link (auth-email-send builds `/auth/callback?token_hash=…&type=signup` → `verifyOtp` → `/dashboard`; `emailRedirectTo` is ignored by the hook, so the dialog's dangling `/auth/confirm` value is inert — see Cross-cutting). Their plan intent is preserved in `user_metadata.planIntent`, so no data loss; they restart checkout from /pricing after confirming. Phase 36 (BILL) lands first and may touch the same pricing-card/stripe-client files — re-verify line anchors after 36. Card unit tests mock `OwnerSubscribeDialog` to `null`, so existing tests are unaffected; the new helper test satisfies the lefthook coverage gate for the changed lines.
- **Files touched:** src/components/pricing/subscribe-complete.ts, src/components/pricing/pricing-card-featured.tsx, src/components/pricing/pricing-card-standard.tsx, src/components/pricing/__tests__/subscribe-complete.test.ts
- **Decision:** Chosen: skip checkout and rely on the dialog's "check your email" toast (smallest correct behavior; no new flows). Alternative: build a resume-checkout flow that reads `user_metadata.planIntent` after email confirmation and auto-opens Stripe checkout post-`/auth/callback` — better funnel, but it spans auth callback + dashboard and belongs to a deliberate BILL/AUTH follow-up, not a UX-defect fix.
