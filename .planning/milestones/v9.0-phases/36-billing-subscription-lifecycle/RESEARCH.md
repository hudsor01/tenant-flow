## BILL-01 — /billing/plans hides portal for delinquent subscribers
- **Finding:** src/app/(owner)/billing/plans/page.tsx:59 (high) — `hasActiveSubscription` accepts only 'active'/'trialing', so a past_due/unpaid owner loses the Manage Subscription button and is routed into `createCheckoutSession`, minting a duplicate subscription.
- **Root cause:** Two conflated concepts. (1) `hasActiveSubscription` is a *gate-status* check, but the page needs a *live-Stripe-subscription* check — past_due/unpaid/paused/incomplete subscriptions still exist on the customer and must be managed via the portal, never via a new checkout. (2) `hasSubscription = currentPlan !== null` couples portal routing to a successful price-id→plan match, so even an *active* subscriber whose `subscription_plan` doesn't match `PRICING_PLANS` (legacy/comp price id — the exact case billing-settings.tsx already Sentry-warns about) would be routed into a second checkout.
- **Fix:** In `page.tsx`: (a) add `const LIVE_SUBSCRIPTION_STATUSES = new Set(["active","trialing","past_due","unpaid","paused","incomplete"])` (comment: mirrors the server-side guard set added in `stripe-checkout/index.ts` for BILL-05 — keep in lockstep); (b) derive `const hasLiveSubscription = status !== null && LIVE_SUBSCRIPTION_STATUSES.has(status) && subscriptionStatus?.stripeCustomerId != null` (the `stripeCustomerId` clause keeps DB-managed trials — trialing with no Stripe customer — on the checkout path, which is correct for them); (c) `hasSubscription = hasLiveSubscription` (decoupled from `currentPlan`); (d) drop the `hasActiveSubscription &&` gate on `currentPlan` so the "Current Plan" info card still renders for a past_due owner. `handleConfirmPlanChange`/`handleManageSubscription` then route all live-subscription users to `createCustomerPortalSession` unchanged.
- **Why it fixes it:** Verifier evidence shows the line-177 portal button hidden and line-95 checkout branch taken for past_due; with `hasLiveSubscription` true for past_due/unpaid, both the button renders and plan clicks take the portal branch, so the duplicate-subscription path is unreachable from this page (server-side backstop is BILL-05).
- **Risks / interactions:** Class fix with BILL-04/BILL-12 (delinquent-lockout class). Trialing-with-Stripe-sub users keep portal routing (correct — plan change via portal). Update `src/app/(owner)/billing/plans` behavior expectations in any unit tests. Phase 44 PUBUX may restyle this page — it rebases on this logic.
- **Files touched:** src/app/(owner)/billing/plans/page.tsx

## BILL-04 — No grace period: past_due locked out with no fix path

Update the middleware test at `src/lib/supabase/__tests__/middleware-routing.test.ts:324` from "redirects past_due to /pricing" to "allows past_due through (grace period)". Align copy in the same phase: banner line 90–91 and the invoice-payment-failed email (BILL-07's file) — replace the hard "7-day" promise with dunning-anchored copy ("update your payment method to avoid losing access" / "before your subscription is canceled") unless the owner pins the Stripe dunning schedule to 7 days in the dashboard (owner-run, out of repo).
- **Why it fixes it:** Verifier evidence: gate bounces every non-active/trialing status while email promises a grace window and the only allowlisted surface hides its portal button. Admitting past_due creates the real grace window AND makes `/settings?tab=billing` (portal launcher) reachable — the in-app path to update the failing card exists again; BILL-01 adds the /billing/plans portal path as belt-and-braces.

RLS/E2E smoke tests that assume past_due→/pricing must be re-checked (`middleware-routing.test.ts` confirmed; grep for other assertions).

## BILL-05 — stripe-checkout has no existing-subscription guard
- **Finding:** supabase/functions/stripe-checkout/index.ts:118 (high) — an already-subscribed owner can complete a second concurrent subscription on the same customer (double billing; webhooks then thrash `users.subscription_*`).
- **Root cause:** The function resolves/creates the customer but never inspects existing subscriptions before `checkout.sessions.create({ mode: "subscription" })`. Client-side guards are absent too (public /pricing cards call it for any session), but the authoritative guard belongs server-side.
- **Fix:** In `stripe-checkout/index.ts`, when `userData.stripe_customer_id` already existed (skip for a just-created customer), call `stripe.subscriptions.list({ customer: customerId, status: "all", limit: 10 })` and filter to live statuses `{active, trialing, past_due, unpaid, paused, incomplete}` (mirror of BILL-01's client set — lockstep comment both sides). If any live subscription exists, do NOT create a checkout session; instead create a billing-portal session (`stripe.billingPortal.sessions.create({ customer: customerId, return_url: `${frontendUrl}/settings?tab=billing` })`) and return 200 `{ url: portalSession.url }`. Every frontend caller already redirects to the returned `url`, so an existing subscriber who clicks Subscribe lands in the portal (the correct plan-change surface) with zero client changes.
- **Why it fixes it:** Verifier evidence: line 118 creates the session with no state check and pricing cards invoke it unguarded. With the guard, the duplicate-subscription branch is unreachable regardless of which client surface calls the function — closing BILL-01's server-side hole for /pricing too.
- **Risks / interactions:** Requires owner-run edge deploy (`bun scripts/deploy-edge-functions.ts`, CLI-401 gotcha). `subscriptions.list` adds ~1 Stripe round-trip for existing customers only. Interacts with BILL-06 (same function, sequence the edits together). Deno tests in `supabase/functions/tests/` may assert the old behavior.
- **Files touched:** supabase/functions/stripe-checkout/index.ts
- **Decision:** Chosen: 200 + portal-session URL (zero client changes, user still lands somewhere actionable). Alternative: 409 `{ error: "existing_subscription" }` and teach each client to route to the portal — more explicit but touches 3+ client surfaces and shows an error toast for a recoverable situation.

## BILL-06 — Every checkout grants a fresh 14-day no-card trial
- **Finding:** supabase/functions/stripe-checkout/index.ts:134 (high) — unconditional `trial_period_days: 14` + `payment_method_collection: "if_required"` + `missing_payment_method: "cancel"` enables infinite serial free trials (expired → checkout → 14 more free days → auto-cancel → repeat).
- **Root cause:** The Stripe trial is granted independently of the DB trial model. The DB already gives every signup one 14-day trial (`set_trial_on_signup`); the checkout should *carry over the remaining DB trial*, never mint a new one. `trial_ends_at` cannot serve as a "trial consumed" marker because `customer-subscription-updated.ts` nulls it for non-trialing statuses.
- **Fix:** In `stripe-checkout/index.ts`: (a) extend the users select (line 88–92) to `"stripe_customer_id, email, full_name, subscription_status, trial_ends_at"`; (b) compute `const remainingTrialMs = userData?.subscription_status === "trialing" && userData?.trial_ends_at ? Date.parse(userData.trial_ends_at) - Date.now() : 0`; (c) only when `remainingTrialMs > 48 * 3600 * 1000` (Stripe requires `trial_end` ≥ 48h out), set `subscription_data.trial_end = Math.floor(Date.parse(userData.trial_ends_at)/1000)` (carry-over, NOT `trial_period_days: 14`) plus `trial_settings.end_behavior.missing_payment_method: "cancel"` and `payment_method_collection: "if_required"`; (d) otherwise omit all three fields — the hosted page collects a card and charges immediately. Delete the now-unused `TRIAL_PERIOD_DAYS` const. Also correct the /billing/plans footer copy ("All plans include a 14-day free trial" → "New accounts include a 14-day free trial") while that file is open for BILL-01/08.
- **Why it fixes it:** Verifier's loop (`expire_trials` → /pricing → new no-card trial → auto-cancel → repeat) breaks at step 3: an 'expired' user has `subscription_status='expired'`, so `remainingTrialMs = 0` and checkout demands payment. A mid-DB-trial converter keeps exactly their remaining trial days instead of a fresh 14 — matching paid tiers' `trial: false` in pricing.ts while honoring the signup promise.
- **Risks / interactions:** Same file as BILL-05/BILL-19 — one coordinated edit + one owner-run deploy. Stripe rejects `trial_end` < 48h away — handled by the threshold. A trialing user with <48h left pays immediately on conversion (acceptable; they clicked Subscribe). Stripe still emits `trial_will_end` for carry-over trials ≥3 days — BILL-20's handler covers notice.
- **Files touched:** supabase/functions/stripe-checkout/index.ts, src/app/(owner)/billing/plans/page.tsx
- **Decision:** Chosen: carry-over of the remaining DB trial via `subscription_data.trial_end`. Alternative: binary "one trial ever" gate on a new persisted marker (e.g. `users.trial_consumed_at`, since `trial_ends_at` is lossy) — rejected: needs a migration + webhook changes for strictly less-correct UX (a day-2 converter would forfeit 12 promised trial days or get 14 fresh ones).

## BILL-14 — No event-ordering guard: stale 'active' can resurrect a canceled subscription
- **Finding:** supabase/functions/stripe-webhooks/handlers/customer-subscription-updated.ts:82 (medium) — the handler writes `sub.status` unconditionally with wall-clock `subscription_updated_at`; a retried older updated(active) processed after deleted(canceled) permanently re-grants access.
- **Root cause:** Ordering key is processing time, not event time. Stripe does not guarantee delivery order and the router deliberately re-processes previously-failed events (index.ts:118), so out-of-order application is a designed-in possibility with no guard.
- **Fix:** Use `event.created` as the ordering key in both subscription handlers. In `customer-subscription-updated.ts` and `customer-subscription-deleted.ts` (Path 2): (a) set `subscription_updated_at: new Date(event.created * 1000).toISOString()` instead of `new Date()`; (b) make the write conditional and atomic via PostgREST filters — `.update(payload).eq("id", user.id).or(\`subscription_updated_at.is.null,subscription_updated_at.lte.${eventCreatedIso}\`)` — so an event older than the persisted watermark is skipped without a read-modify-write race; (c) `logEvent` a "skipped stale event" line when the update matches 0 rows (request `select("id")` on the update to observe row count). `checkout-session-completed.ts` also switches to `event.created` for consistency of the watermark column.
- **Why it fixes it:** Verifier's concrete path — failed updated(active, created T1) retried after deleted(canceled, created T2>T1) — now no-ops: deleted stamped the watermark at T2, the retry's T1 fails the `lte` filter, 'canceled' stands. `lte` (not `lt`) keeps same-event replays and same-second successors idempotent/applied.
- **Risks / interactions:** Any pre-existing rows with future-skewed `subscription_updated_at` (wall-clock writes are ≥ event time, skew is seconds) could reject the first post-deploy event of the same second — harmless, next event applies. Same files as BILL-13; single owner-run deploy. Deno webhook tests should add an out-of-order case.
- **Files touched:** supabase/functions/stripe-webhooks/handlers/customer-subscription-updated.ts, supabase/functions/stripe-webhooks/handlers/customer-subscription-deleted.ts, supabase/functions/stripe-webhooks/handlers/checkout-session-completed.ts

## BILL-15 — Events stuck in 'processing' are acked as duplicates forever
- **Finding:** supabase/functions/stripe-webhooks/index.ts:118 (medium) — on 23505 duplicate, only `status === 'failed'` is re-marked; a row orphaned in 'processing' (isolate kill between insert and the status update) makes every Stripe retry 200 and the event is silently lost.
- **Root cause:** The idempotency state machine has no recovery transition out of 'processing'; the insert-first design guarantees orphans whenever the runtime dies mid-handler.
- **Fix:** In the duplicate branch, select `status, processed_at` (not just `status`); treat as reprocessable when `existing.status === "failed"` OR (`existing.status === "processing"` AND `processed_at` older than a 5-minute staleness threshold — Edge Function wall-clock max is well under that, so an old 'processing' row is provably dead). On re-mark, also refresh `processed_at: new Date().toISOString()` so concurrent retries in the same window don't both claim it. Define `const STALE_PROCESSING_MS = 5 * 60_000` with a comment tying it to the runtime ceiling.
- **Why it fixes it:** Verifier confirmed lines 164–179 are the only status transitions and no cron/cleanup touches 'processing'. The staleness transition gives Stripe's 72-hour retry machinery a path to redeliver the lost event (e.g. a deleted that must revoke access), and BILL-14's ordering guard makes the re-run safe.
- **Risks / interactions:** A theoretical double-run if two retries race the 5-minute boundary — subscription writes are idempotent/ordered post-BILL-14; `invoice.payment_failed` could double-send one email in that rare race (accepted; Stripe retry cadence makes simultaneous retries unlikely). Same file as BILL-20's router change; one deploy.
- **Files touched:** supabase/functions/stripe-webhooks/index.ts
