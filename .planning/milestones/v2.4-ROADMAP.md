# Roadmap: TenantFlow v2.4 — Document Vault Extension + Category Taxonomy

**Defined:** 2026-04-22
**Target ship:** 2026-05-06 (2 weeks, 3 phases)
**Scope:** Close the document-vault deferred list from v2.3 — extend the property-only vault to lease, tenant, and maintenance entities; add a global `/documents/vault` with search + filter; replace the free-form `document_type` column with a categorical taxonomy. Do nothing else.

## Why This Milestone Exists

v2.3 Phase 57 shipped the document vault for **property-scoped** uploads. The backing table (`public.documents`) already supports `entity_type IN ('property', 'lease', 'tenant', 'maintenance_request', 'inspection')` — that's a pre-existing schema choice — but the UI surface and storage-RLS branches only ship the property branch today.

Three concrete pain points remain:

1. **Owners can't attach documents to non-property entities.** Avail/Buildium reviewers cite "can't find documents I uploaded" as recurring. v2.3 addressed the biggest bucket (property docs); leases/tenants/maintenance remain unreachable. Every lease + tenant + inspection + maintenance record generated by the v2.3 bulk-import feature would benefit from having associated documents.

2. **No global view.** A user looking for "that receipt from March" has to remember which property it was attached to. Needs a `/documents/vault` page with filtering + search.

3. **`document_type` is MIME-shaped instead of categorical.** v2.3's cycle-4 audit added a separate `mime_type` column but left `document_type` with a default of `'other'` and no enforced taxonomy. Owners can't answer "show me all my tax documents" today. A real taxonomy (`lease`, `receipt`, `tax_return`, `inspection_report`, `maintenance_invoice`, `insurance`, `other`) unlocks filter UI.

Bulk-import polish (column mapping UI, Buildium/AppFolio format detection) is intentionally deferred to v2.5+ — the bulk-import deferred list is less user-visible than the vault gap.

## Success Criteria

- On `/leases/[id]`, `/tenants/[id]`, and `/maintenance/[id]` an owner can upload PDFs/images and see them listed, reusing the exact components shipped in v2.3 (`DocumentsSection`, `DocumentRow`, signed-URL batching, storage-RLS path validation).
- On `/documents` (new route) an owner sees all their documents across all entity types in one list, with filters for entity type, category, and a search box that hits titles + descriptions + tags.
- `documents.document_type` enforces a taxonomy via CHECK constraint + Zod enum; upload dialog lets the user pick a category (or accept the default `'other'`).
- No regressions across v2.0 gates, v2.2 widgets, v2.3 bulk-import, or the `lease_tenants` RLS policies added 2026-04-22.

## Phases (strict ordering — 60 reuses 59's RLS pattern, 61 gates on 60's taxonomy)

- [ ] **Phase 59: Storage RLS + upload entry points for lease/tenant/maintenance** — extends `tenant-documents` bucket policies to the three new entity types and renders `<DocumentsSection>` on each detail page.
- [ ] **Phase 60: Global `/documents/vault` page with search + filter** — new route, new hook, reuses signed-URL batching.
- [ ] **Phase 61: Category taxonomy + filter UI** — add CHECK constraint on `document_type`, rewire upload dialog to select a category, wire category filter into the global vault.

## Phase 59: Storage RLS + Upload Entry Points for Lease/Tenant/Maintenance

**Goal:** An owner viewing a lease/tenant/maintenance record can attach + preview + delete documents on that record using the exact Phase 57 UI surface.

**Depends on:** v2.3 Phase 57 shipped (migrations `20260420030000`, `20260421120000`, `20260422120000`, `20260422130000`, `20260423120000`, `20260424120000`).

**Audit findings driving the scope:**
- `public.documents` already supports all four entity types per migration `20260306140000_documents_owner_column.sql:26-50`. No schema changes needed.
- `cleanup_orphan_documents` was extended to cover all four entity types in migration `20260422120000`. No change needed.
- `tenant-documents` bucket's storage RLS has a single `or` clause for `property` today. Extension needs three more `or` clauses matching the same `owner_user_id` join pattern through the parent table (`leases`, `tenants`, `maintenance_requests`).
- `documentQueries.list({ entityType, entityId })` already accepts `entityType` as a generic parameter; the type `DocumentEntityType = 'property'` needs to widen to the full union.

**Requirements:**

- **REQ-59-01**: Migration extends the three storage RLS policies (`"Owners upload tenant documents"`, `"Owners view tenant documents"`, `"Owners delete tenant documents"`) to cover `lease`, `tenant`, and `maintenance_request` branches. Each branch joins through its parent table to `owner_user_id = (select auth.uid())`. Keep the `array_length = 2` and UUID-format guards.

- **REQ-59-02**: `DocumentEntityType` widens from `'property'` to `'property' | 'lease' | 'tenant' | 'maintenance_request'`. Every call site (there is one in `documents-section.tsx`) still passes `'property'` until explicitly widened.

- **REQ-59-03**: Render `<DocumentsSection entityType="lease" entityId={lease.id} />` on `/leases/[id]` between the Lease Summary card and the rent-due table. Same placement pattern on `/tenants/[id]` and `/maintenance/[id]`.

- **REQ-59-04**: Integration test: `tests/integration/rls/documents-cross-entity.rls.test.ts` exercises cross-owner isolation on each branch — ownerA uploads to their lease, ownerB cannot SELECT that document.

- **REQ-59-05**: Unit test updates: `documents-section.test.tsx` parameterizes its test data across all four entity types (factored via `describe.each`).

**Battle-proven criterion:** on `/leases/<id>`, `/tenants/<id>`, `/maintenance/<id>` for three different records owned by three different owners, each owner can upload a 3 MB PDF to their own record, see it listed, preview it, and delete it. Owner B gets 0 rows on ownerA's records (verified via dual-client test).

**Deferred to v2.5:**
- Inspection branch (fifth entity type) — shipping 4 of 5 is still a huge UX win; inspection uploads are niche.
- Bulk upload multi-file picker on non-property entities (v2.3 only implemented the multi-file pattern for property).

## Phase 60: Global `/documents/vault` Page with Search + Filter

**Goal:** An owner visits `/documents` and sees all their documents across all entity types in one virtual-scrolled list, with filters for entity type + category (category filter wires up in Phase 61) + a search box that hits titles + descriptions + tags.

**Depends on:** Phase 59 shipped (so the global list actually contains something besides property docs).

**Audit findings driving the scope:**
- No existing `/documents` route.
- `tenant-documents` bucket RLS already correctly filters to rows where the owner owns the parent entity. The global-vault query can do a plain `select * from documents where owner_user_id = auth.uid()` — RLS already enforces it, and `owner_user_id` was backfilled on all existing rows via migration `20260306140000`.
- Full-text search: use the `tsvector` column + `to_tsvector(unaccent(coalesce(title,'') || ' ' || coalesce(description,'') || ' ' || coalesce(array_to_string(tags,' '),'')))` pattern. Generated column + GIN index. Matches the pattern used in `get_blog_posts` already.

**Requirements:**

- **REQ-60-01**: Migration adds a generated `search_vector tsvector` column on `public.documents` driven by `title + description + tags`. Add `GIN index` on it. Backfill is automatic (generated column).

- **REQ-60-02**: New RPC `search_documents(p_query text default null, p_entity_type text default null, p_category text default null, p_limit int default 50, p_offset int default 0)` returns `documents` rows filtered by owner (via RLS) + optional query/entity/category. Uses `search_vector @@ plainto_tsquery(p_query)` when `p_query` is non-empty.

- **REQ-60-03**: New hook `documentSearch.list(filters)` in `src/hooks/api/query-keys/document-keys.ts` calls the RPC, batches `createSignedUrls` for the returned rows. Same 45min staleTime / 55min gcTime pattern as `documentQueries.list`.

- **REQ-60-04**: New page `/app/(owner)/documents/page.tsx` with a search input (debounced 300ms), entity-type filter dropdown, virtualized list (reuse `@tanstack/react-virtual` pattern from property list), preview dialog (reuse `<DocumentRow>` from Phase 57).

- **REQ-60-05**: Nav link: add "Documents" to the owner sidebar (between "Maintenance" and "Reports") with the `FolderArchive` lucide icon.

- **REQ-60-06**: Integration tests: 6 cases on the RPC — owner isolation, empty query returns all, text search matches title/description/tags, entity-type filter, category filter, pagination.

**Battle-proven criterion:** an owner with 12 property docs + 8 lease docs + 5 maintenance docs navigates to `/documents`, sees 25 rows. Typing "tax" in the search box filters to rows where title or tags contain "tax". Selecting "Lease" in the entity dropdown filters to the 8 lease docs.

**Deferred to v2.5:**
- Date-range filter (created_at between X and Y)
- Multi-select filter (pick multiple categories at once)
- Bulk download as zip (needs Edge Function + ZipStream; non-trivial)
- Bulk delete from the vault page (low user value given per-row delete works)

## Phase 61: Category Taxonomy + Filter UI

**Goal:** Replace `document_type`'s default `'other'` with a fixed categorical enum. Owners select a category when uploading (default `'other'`); the vault page filters by category.

**Depends on:** Phase 60 shipped (so the filter has a list to filter).

**Audit findings driving the scope:**
- `document_type` is currently a freetext column with default `'other'`. Cycle-2 audit flagged that v2.3 was cramming MIME-type values into it; cycle-4 fix added a separate `mime_type` column but left `document_type` as freetext.
- Reasonable categories for landlord use: `lease`, `receipt`, `tax_return`, `inspection_report`, `maintenance_invoice`, `insurance`, `other`.
- Competitor parity: Avail, Buildium, and TurboTenant all have category filtering. TenantFlow doesn't.

**Requirements:**

- **REQ-61-01**: Migration adds CHECK constraint `document_type IN ('lease','receipt','tax_return','inspection_report','maintenance_invoice','insurance','other')`. Any non-conforming existing row (there should be zero given v2.3 defaults) rewrites to `'other'` in the same migration.

- **REQ-61-02**: Zod enum `documentCategorySchema` in `src/lib/validation/documents.ts` matching the CHECK values. Export `DOCUMENT_CATEGORIES` tuple for UI reuse (same pattern as `TENANT_ACTIVE_STATUSES`).

- **REQ-61-03**: Upload dialog gains a `<Select>` for category (defaults to `'other'`). Passed to `documentMutations.upload` via the existing `category` param (already in the type; wires up now).

- **REQ-61-04**: Global vault filter dropdown `Category: [Any / Lease / Receipt / ...]`. Wires into `documentSearch.list` via the already-designed `p_category` RPC parameter.

- **REQ-61-05**: Unit + integration tests updated: Zod enum tested, RPC tested with category filter.

**Battle-proven criterion:** an owner uploads 3 documents to a property — one tagged `tax_return`, one tagged `insurance`, one accepts default `other`. On `/documents`, selecting category `tax_return` shows only the first. Selecting `Any` shows all three.

**Deferred to v2.5:**
- Custom user-defined categories (let owners add `'warranty'`, `'repair_quote'`, etc.). Requires a `document_categories` table + admin UI.
- Auto-categorization via keyword heuristics (e.g., filename "W-2" → `tax_return`).
- Category-scoped permissions (share specific categories with a CPA). Requires a sharing model.

## Sequencing Rules

1. Phase 59 ships first. It's the foundation — global vault has nothing to show until non-property uploads work.
2. Phase 60 depends on Phase 59's data. Ship the global vault once the entity-type dimension has real data.
3. Phase 61 depends on Phase 60's filter infrastructure. Ship category last so the enum + UI + RPC align in one PR.
4. Each phase ships its own PR. No bundling. Each PR includes an integration test that gates the CI `rls-security` check.
5. Post-merge: no Edge Function deploy needed (everything is PostgREST + RPCs).

## Out of Scope

- Bulk-import polish (column mapping, Buildium/AppFolio auto-detect) — defer to v2.5. The existing v2.3 importer is usable.
- Document versioning / revision history — niche, defer.
- Cloud storage integration (Google Drive, Dropbox) — high eng cost, low user value.
- Public tenant-facing document view — tenants are records in landlord-only mode, not authenticated users.
- OCR / text extraction — future premium-tier feature.
- Inspection branch of the document vault — 4-of-5 is enough for v2.4; inspections are niche and can wait.

## Revenue Impact

Retention-driven, not conversion-driven. Same shape as v2.3. Expected effect: reduce churn on 14-day-trial users who start uploading docs to one property, realize they can't attach docs to the lease itself, and bounce. Harder to measure than conversion gates; defers to a post-v2.4 funnel analysis once 20+ paid users have had a month with the feature.

## Risks

- **Storage RLS complexity** (Phase 59): four `or` clauses with four distinct joins is more error-prone than one. Integration tests on every entity type are mandatory; the cycle-4 `lease_tenants` gap proved RLS-policy gaps can go undetected for months.
- **Full-text search correctness** (Phase 60): generated `tsvector` columns trigger on every row update. Usually benign, but worth benchmarking once at 500 rows to confirm no N+1 pattern leaks into the UI.
- **`document_type` CHECK constraint rollout** (Phase 61): if any existing row's value isn't in the enum, the ADD CONSTRAINT fails. Migration includes a defensive `UPDATE ... SET document_type = 'other' WHERE document_type NOT IN (...)` before the ALTER TABLE.
