# Roadmap: TenantFlow v2.1 — Production Integrity Hardening

**Defined:** 2026-04-19
**Target ship:** 2026-04-26 (1 week, critical fixes only)
**Pauses:** v2.0 Phase 46 (premium reports gate) — do not ship more paywalls until the product behind them actually works

## Why This Milestone Exists

A 2026-04-19 prod audit found the MVP is broken in at least four ways. Integration tests pass trivially because they use `forEach` over empty arrays — a misdesign that masks a non-functional product. v2.0 Phase 45 shipped conversion-tracking for an e-sign gate, but the underlying landlord dashboard cannot read its own data, and the admin page Phase 45 wired up crashes when rendered. Shipping a second paywall (Phase 46) on top of this is negligent.

**Success criterion for the milestone as a whole:** a real seeded landlord opens `/dashboard` and sees their own properties, units, leases, tenants, and maintenance requests. Billing page reflects their real Stripe subscription state. Admin page renders `get_gate_conversion_stats` without 500ing. Stripe zombie Edge Functions deleted. Test suite fails when RLS blocks legitimate reads.

Everything here is recovery from drift caused by PR #596 (landlord-only pivot) and prior migrations. No new feature work.

## Phase 47: RLS Recovery + Test Suite Correctness

**Goal:** `/dashboard`, `/leases`, `/maintenance`, `/properties/:id` all render real data for a seeded landlord. Tests actively fail when they don't.

**Requirements:**
- **REQ-47-01**: Restore RLS policies on `public.leases` (owner_user_id = auth.uid() pattern per CLAUDE.md § RLS Rules)
- **REQ-47-02**: Restore RLS policies on `public.maintenance_requests` (same pattern)
- **REQ-47-03**: Restore RLS policies on `public.notifications` (user_id = auth.uid())
- **REQ-47-04**: Integration tests for leases/maintenance/notifications add an assertion that authenticated owner reads AT LEAST ONE row of seeded data. Empty array fails the test explicitly. This is the pattern that would have caught the drift.
- **REQ-47-05**: Audit the 11 other RLS-no-policy tables — each either gets proper policies OR documented service-role-only comment explaining why authenticated roles shouldn't touch it
- **REQ-47-06**: Verify by signing in as seeded owner and hitting `/dashboard` — confirm real rows appear

**Success gate:** `pnpm test:integration` fails if any core table returns empty for a seeded owner. `/dashboard` on prod shows seeded data.

## Phase 48: Zombie Edge Function Purge

**Goal:** Remove attack surface from removed-feature Edge Functions that are still accepting traffic in prod.

**Requirements:**
- **REQ-48-01**: Delete prod Edge Functions via MCP: `stripe-connect`, `stripe-rent-checkout`, `stripe-autopay-charge`, `detach-payment-method`, `tenant-invitation-validate`, `tenant-invitation-accept`, `send-tenant-invitation`, `stripe-setup`
- **REQ-48-02**: Delete local source directories for the same functions under `supabase/functions/`
- **REQ-48-03**: Remove any `[functions.<name>]` entries in `supabase/config.toml`
- **REQ-48-04**: Grep codebase for any lingering `supabase.functions.invoke('<name>', ...)` calls and remove / fix

**Success gate:** `supabase functions list` (or MCP list_edge_functions) shows only the active set. No invocations in code target deleted functions.

## Phase 49: Billing Path + Stripe Sync Engine Restoration

**Goal:** Billing UI reflects real Stripe state. `get_gate_conversion_stats` stops 500ing. Any new subscription / invoice / customer update from Stripe lands in the DB within seconds.

**Requirements:**
- **REQ-49-01**: Verify `get_subscription_status` RPC — either (a) it exists and the audit query missed it (re-check `information_schema.routines`), or (b) it was dropped and needs restoration from migration history
- **REQ-49-02**: Diagnose why Stripe Sync Engine only has `stripe.accounts` and `stripe.*` engine-internal tables. Expected: `stripe.subscriptions`, `stripe.customers`, `stripe.invoices`, `stripe.checkout_sessions`, `stripe.prices`, `stripe.products`. Fix: either configure the sync to include these objects OR remove the Sync Engine and read from `users.subscription_*` columns (already populated by stripe-webhooks handler)
- **REQ-49-03**: Rewrite `get_gate_conversion_stats` to NOT depend on `stripe.subscriptions`. Either join against `users.subscription_plan` + `users.subscription_status` (already in the table) with a manually-tagged conversion path, OR write a new `gate_conversions` table that the stripe-webhooks handler inserts into on `customer.subscription.created` when `subscription.metadata.source` is set
- **REQ-49-04**: Verify `/billing/plans`, `/settings?tab=billing`, `/admin/analytics` all render without 500 for a real signed-in user (not just typecheck pass)

**Success gate:** admin page renders the full Paywall Conversions + Email Deliverability + Funnel tables. Billing UI shows current plan + renewal date for a seeded owner with a test-mode subscription.

## Phase 50: Security Advisor Cleanup

**Goal:** Supabase security advisor returns zero WARN+ findings for the project.

**Requirements:**
- **REQ-50-01**: Enable Supabase Auth leaked-password protection (HaveIBeenPwned check) in Dashboard → Authentication → Password Security
- **REQ-50-02**: Tighten storage RLS for `avatars` + `property-images` buckets — SELECT policy should allow access to specific object paths, not list-all. Pattern: `(storage.foldername(name))[1] = (select auth.uid())::text` for user-scoped assets; bucket stays public for object-URL reads but directory listing is removed
- **REQ-50-03**: Add `SET search_path = ''` to 3 stripe schema functions flagged: `stripe.set_updated_at_metadata`, `stripe.set_updated_at`, `stripe.check_rate_limit`
- **REQ-50-04**: Final `get_advisors` run — zero WARN-or-higher findings

**Success gate:** `mcp__plugin_supabase_supabase__get_advisors` returns only INFO findings (or nothing).
