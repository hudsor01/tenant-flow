inspection-photo-upload.tsx (CURRENT bugs):
- FileUploadState = { file, objectUrl, status: "pending"|"uploading"|"success"|"error", error? } — no stable id
- onDrop pushes files onto `files` (index in `files` != index in the later `pendingFiles` filter)
- handleUpload (line ~68): `pendingFiles = files.filter(f => f.status==="pending")`, then
  `pendingFiles.map(async (fileState, idx) => { … setFiles(prev => prev.map((f,i)=> i===idx ? … )) })`
  → `idx` is the filtered index but is applied to the FULL `files` array → wrong tile updated
- allSettled rejection path never sets status "error" (the catch is implicit; only success/uploading set)
- no guard prevents re-running upload on a file already "success" (only "pending" is filtered in, but a
  retry design must still explicitly skip "success")
- removeFile(index) uses the array index (fine for the render map, but tie removal to the stable id too)

useRecordInspectionPhoto(inspectionId) mutation records the DB row; a duplicate call = duplicate row.
Upload uses upsert:false to storage, so a duplicate storage path throws — the DB duplicate is the real risk.
